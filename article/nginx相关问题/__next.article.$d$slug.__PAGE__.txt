<h3 id="第一层基础概念考察基本功和知识广度"><a href="#第一层基础概念考察基本功和知识广度">第一层：基础概念（考察基本功和知识广度）</a></h3>
<p>这类问题用于快速筛选，确认你是否具备最基本的 Web 服务器知识。</p>
<h4 id="问题-1请你讲讲-nginx-是什么它主要用来做什么"><a href="#问题-1请你讲讲-nginx-是什么它主要用来做什么">问题 1：请你讲讲 Nginx 是什么？它主要用来做什么？</a></h4>
<p><strong>面试官意图：</strong> 确认你是否知道 Nginx 的定位和核心功能，而不是把它当成一个黑盒。</p>
<p><strong>回答思路：</strong></p>
<ol>
<li>
<p><strong>定性：</strong> 首先，明确 Nginx 是一款<strong>高性能</strong>的 <strong>Web 服务器</strong>和<strong>反向代理服务器</strong>。它的关键特性是<strong>异步、事件驱动</strong>，这使得它在处理高并发连接时，相比传统的 Apache 等服务器，占用内存更少，性能更高。</p>
</li>
<li>
<p><strong>核心功能（至少说出三点）：</strong></p>
<ul>
<li>
<p><strong>静态资源服务器：</strong> 这是最基本的功能。可以直接将服务器上的静态文件（如 HTML、CSS、JavaScript、图片等）高效地返回给客户端。对于前后端分离的项目，打包后的前端静态资源通常就由 Nginx 托管。</p>
</li>
<li>
<p><strong>反向代理 (Reverse Proxy)：</strong> 这是 Nginx 最核心、最强大的功能。客户端请求 Nginx，Nginx 再根据配置将请求转发给后端的应用服务器（如 Node.js、Tomcat、Python 应用等），然后将后端服务器的响应再返回给客户端。这个过程对客户端是透明的，客户端以为自己一直在和 Nginx 通信。它能实现负载均衡、隐藏后端服务细节等。</p>
</li>
<li>
<p><strong>负载均衡 (Load Balancing)：</strong> 当后端有多台应用服务器时，Nginx 可以作为流量入口，将客户端的请求按照指定策略（如轮询、权重、IP Hash 等）分发到不同的服务器上，从而实现服务的水平扩展和高可用。</p>
</li>
<li>
<p><strong>API 网关 (API Gateway)：</strong> 在微服务架构中，Nginx 可以作为 API 网关，提供统一的入口，负责鉴权、路由、日志、监控、限流等公共功能。</p>
</li>
</ul>
</li>
</ol>
<p><strong>深入讲解：</strong></p>
<p>可以进一步对比 Nginx 和 Apache 的区别来突显你的理解深度。</p>
<ul>
<li>
<p><strong>工作模型：</strong> Nginx 使用异步非阻塞的事件驱动模型（epoll/kqueue），一个工作进程（worker process）可以处理成千上万个连接，内存占用极低。而 Apache 默认使用多进程/多线程模型，每个连接会占用一个进程或线程，当并发量上来时，内存开销巨大。</p>
</li>
<li>
<p><strong>功能侧重：</strong> Nginx 对静态资源的处理和反向代理性能极佳，但对动态内容需要通过 FastCGI 等方式转发给后端应用处理。Apache 功能更全面，对动态内容（如 PHP）的支持更“原生”，但整体性能和并发能力不如 Nginx。</p>
</li>
</ul>
<p><strong>一句话总结：Nginx 就像一个高效的交通枢纽，既能快速地把静态货物（静态资源）直接发走，也能作为总调度室（反向代理），把各种复杂的请求（API 请求）精准地派发给后面不同的处理部门（应用服务器），还能确保没有哪个部门过于劳累（负载均衡）。</strong></p>

<h4 id="问题-2我们用-vuereact-做了个单页面应用spa用-nginx-部署时需要注意什么配置怎么写"><a href="#问题-2我们用-vuereact-做了个单页面应用spa用-nginx-部署时需要注意什么配置怎么写">问题 2：我们用 Vue/React 做了个单页面应用（SPA），用 Nginx 部署时需要注意什么？配置怎么写？</a></h4>
<p><strong>面试官意图：</strong> 这是前端部署的核心场景。考察你是否理解 SPA 的路由原理，以及如何通过 Nginx 解决 <code>history</code> 模式下的 404 问题。</p>
<p><strong>回答思路：</strong></p>
<ol>
<li>
<p><strong>点明问题核心：</strong> SPA 应用使用前端路由（如 Vue Router 或 React Router）的 <code>history</code> 模式时，URL 的改变是通过 <code>history.pushState()</code> API 实现的，浏览器并不会真的向服务器发起请求。但当用户在某个路由下（例如 <code>/user/profile</code>）刷新页面，或者直接访问这个 URL 时，浏览器会向服务器请求 <code>/user/profile</code> 这个资源。然而，在我们的静态资源目录（通常是 <code>dist</code> 或 <code>build</code>）下，并不存在 <code>user/profile</code> 这个文件或目录，只有一个 <code>index.html</code>。因此，Nginx 默认会返回 404 Not Found。</p>
</li>
<li>
<p><strong>给出解决方案：</strong> 核心解决方案是：<strong>无论用户请求什么路径，只要这个路径在服务器上找不到对应的静态文件，就都让他返回主入口文件 <code>index.html</code></strong>。剩下的路由工作，交给 <code>index.html</code> 中加载的 JavaScript（即前端路由库）来处理。</p>
</li>
<li>
<p><strong>展示具体配置：</strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx">server {
    listen       80;
    server_name  your.domain.com;

    # 指定前端打包后的静态资源目录
    root         /var/www/html/your-project/dist;

    # 指定入口文件
    index        index.html index.htm;

    # 核心配置
    location / {
        try_files $uri $uri/ /index.html;
    }

    # 其他配置，例如 API 代理
    location /api/ {
        proxy_pass http://backend-api-server;
    }
}
</code></pre>
</li>
<li>
<p>彻底讲清楚 try_files：</p>
<p>try_files <span class="katex"><span class="katex-mathml"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>u</mi><mi>r</mi><mi>i</mi></mrow><annotation encoding="application/x-tex">uri </annotation></semantics></math></span><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.6595em;"></span><span class="mord mathnormal">u</span><span class="mord mathnormal" style="margin-right:0.02778em;">r</span><span class="mord mathnormal">i</span></span></span></span>uri/ /index.html; 是这里的关键。它的工作逻辑是：</p>
<ul>
<li>
<p><code>$uri</code>：尝试查找与用户请求 URL 完全匹配的文件。例如，请求 <code>/assets/logo.png</code>，就去 <code>root</code> 目录下查找 <code>/assets/logo.png</code> 文件。</p>
</li>
<li>
<p><code>$uri/</code>：如果上一步没找到，就尝试查找与 URL 匹配的目录。例如，请求 <code>/user/</code>，就看是否存在 <code>/user/</code> 目录。</p>
</li>
<li>
<p><code>/index.html</code>：如果前两者都找不到，就会内部重定向到 <code>/index.html</code>。此时，浏览器 URL 依然是 <code>/user/profile</code>，但服务器实际返回的是 <code>index.html</code> 的内容。前端的路由库一旦加载，就会读取 URL，并渲染出 <code>/user/profile</code> 对应的组件。</p>
</li>
</ul>
</li>
</ol>
<p>这个问题回答得好坏，直接体现了你是否亲手部署过项目。</p>
<h4 id="问题-3开发环境中如何用-nginx-解决跨域问题"><a href="#问题-3开发环境中如何用-nginx-解决跨域问题">问题 3：开发环境中，如何用 Nginx 解决跨域问题？</a></h4>
<p><strong>面试官意图：</strong> 考察你对同源策略的理解，以及利用反向代理解决实际开发痛点的能力。</p>
<p><strong>回答思路：</strong></p>
<ol>
<li>
<p><strong>阐述问题根源：</strong> 首先解释什么是跨域。浏览器出于安全考虑，实行<strong>同源策略</strong>，即协议（protocol）、域名（host）、端口（port）三者完全一致才算同源。开发时，前端项目跑在 <code>http://localhost:8080</code>，而后端 API 服务在 <code>http://api.example.com</code>，端口或域名不同，直接在前端代码里发请求就会被浏览器拦截，产生跨域错误。</p>
</li>
<li>
<p><strong>提出解决方案（Nginx 反向代理）：</strong> 我们可以设置一个 Nginx 服务器，它同时服务我们的前端静态页面，并把 API 请求代理转发到后端服务器。</p>
<ul>
<li>
<p><strong>对浏览器而言：</strong> 浏览器所有请求都发往 Nginx 所在的地址（例如 <code>http://localhost:80</code>）。无论是请求页面 <code>http://localhost/</code> 还是请求数据 <code>http://localhost/api/users</code>，都是同源的，因此浏览器不会拦截。</p>
</li>
<li>
<p><strong>对 Nginx 而言：</strong> Nginx 内部进行判断。如果请求的是页面，就返回静态文件。如果请求的是以 <code>/api/</code> 开头的路径，就通过 <code>proxy_pass</code> 指令将请求转发给真正的后端 API 服务器。</p>
</li>
</ul>
</li>
<li>
<p><strong>展示具体配置：</strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx">server {
    listen       80;
    server_name  localhost;

    # 前端静态资源
    location / {
        root   /path/to/your/frontend/dist;
        index  index.html;
        try_files $uri $uri/ /index.html;
    }

    # API 代理配置
    location /api/ {
        # 转发到后端 API 服务器地址
        proxy_pass   http://api.example.com/;

        # 可以重写请求头，传递真实信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
</code></pre>
</li>
<li>
<p><strong>深入讲解细节：</strong></p>
<ul>
<li>
<p><strong><code>proxy_pass</code> 的斜杠问题</strong>：这是一个常见的坑，能讲出来是加分项。</p>
<ul>
<li>
<p><code>proxy_pass http://api.example.com;</code> (带路径)</p>
</li>
<li>
<p><code>proxy_pass http://api.example.com/;</code> (不带路径)</p>
</li>
<li>
<p><strong>规则</strong>：如果 <code>proxy_pass</code> 的地址<strong>包含路径</strong>（如 <code>/</code>），那么转发时会用这个路径替换掉 <code>location</code> 匹配到的部分。如果<strong>不包含路径</strong>，则会直接将 <code>location</code> 匹配到的路径拼接到后面。</p>
</li>
<li>
<p><strong>举例</strong>：对于 <code>location /api/ { ... }</code>，请求 <code>/api/users</code></p>
<ul>
<li>
<p><code>proxy_pass http://backend/v1/</code> -> 转发到 <code>http://backend/v1/users</code></p>
</li>
<li>
<p><code>proxy_pass http://backend;</code> -> 转发到 <code>http://backend/api/users</code></p>
</li>
</ul>
</li>
</ul>
</li>
<li>
<p>除了 Nginx 代理，也可以简单提一下其他开发环境的跨域解决方案，如 Webpack 的 <code>devServer.proxy</code> 或 Vite 的 <code>server.proxy</code>，并说明它们的底层原理其实也是启动了一个类似的代理服务器。这能体现你的知识关联能力。</p>
</li>
</ul>
</li>
</ol>

<h4 id="问题-4如何用-nginx-来优化前端页面的加载速度"><a href="#问题-4如何用-nginx-来优化前端页面的加载速度">问题 4：如何用 Nginx 来优化前端页面的加载速度？</a></h4>
<p><strong>面试官意图：</strong> 这是一个开放性问题，考察你对前端性能优化手段在 Nginx 层面如何落地的理解。</p>
<p><strong>回答思路：</strong> 这是一个组合拳，需要从多个角度来回答。</p>
<ol>
<li>
<p><strong>开启 Gzip 压缩：</strong></p>
<ul>
<li>
<p><strong>原理：</strong> Gzip 可以对文本类文件（HTML, CSS, JS, JSON, SVG等）进行压缩，显著减小文件体积，从而减少网络传输时间。图片（JPG, PNG）和视频等因为本身已经压缩过，再用 Gzip 效果不佳，反而浪费 CPU。</p>
</li>
<li>
<p><strong>配置：</strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx">http {
    # 开启 gzip
    gzip on;

    # 小于 1k 的文件不压缩，因为压缩本身有开销
    gzip_min_length 1k;

    # 压缩级别，1-9，级别越高压缩率越高但越耗 CPU，通常设为 4-6
    gzip_comp_level 6;

    # 指定要压缩的文件类型
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript image/svg+xml;

    # 告诉代理服务器也要压缩
    gzip_proxied any;
}
</code></pre>
</li>
</ul>
</li>
<li>
<p><strong>配置浏览器缓存 (Browser Caching)：</strong></p>
<ul>
<li>
<p><strong>原理：</strong> 通过设置 HTTP 响应头中的 <code>Cache-Control</code> 或 <code>Expires</code>，告诉浏览器在多长时间内可以直接使用本地缓存的副本，而无需再次向服务器请求。这对于不经常变化的静态资源（如带 hash 的 JS/CSS 文件、logo 图片）尤其有效。</p>
</li>
<li>
<p><strong>配置：</strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx"># 对图片、JS、CSS 等不常变动的文件设置长缓存
location ~* \.(js|css|jpg|jpeg|png|gif|svg|ico)$ {
    # 设置过期时间为 30 天
    expires 30d;
    # 也可以用 add_header 设置 Cache-Control
    # add_header Cache-Control "public, max-age=2592000";
}

# 对 HTML 文件通常不建议设置长缓存，或者设置 no-cache
# 因为 HTML 是入口文件，需要保证用户能及时获取到最新的版本
location ~* \.html$ {
    # add_header Cache-Control "no-cache, must-revalidate";
    expires -1; # 或者设置为不缓存
}
</code></pre>
</li>
<li>
<p><strong>引申：</strong> 这里一定要提到**缓存失效（Cache Busting）**策略。解释为什么我们可以对 JS/CSS 设置长缓存——因为现代前端构建工具（Webpack/Vite）会在文件名中加入 hash 值（如 <code>app.d2a1b3.js</code>）。一旦文件内容改变，hash 就会改变，文件名也就变了，浏览器自然会去请求新的文件，完美解决了更新问题。而 <code>index.html</code> 因为文件名不变，所以不能设置长缓存。</p>
</li>
</ul>
</li>
<li>
<p><strong>HTTP/2 支持：</strong></p>
<ul>
<li>
<p><strong>原理：</strong> HTTP/2 相比 HTTP/1.1 有巨大性能提升，主要体现在：</p>
<ul>
<li>
<p><strong>多路复用 (Multiplexing)：</strong> 多个请求可以在一个 TCP 连接上并行进行，解决了 HTTP/1.1 的队头阻塞问题。</p>
</li>
<li>
<p><strong>头部压缩 (Header Compression)：</strong> 使用 HPACK 算法压缩请求头，减少了传输开销。</p>
</li>
<li>
<p><strong>服务器推送 (Server Push)：</strong> 服务器可以主动将资源推送到客户端缓存中。</p>
</li>
</ul>
</li>
<li>
<p><strong>配置：</strong> 只需要在 <code>listen</code> 指令后加上 <code>http2</code> 即可。当然，前提是需要先配置好 HTTPS。</p>
<p>Nginx</p>
<pre><code class="hljs language-nginx">server {
    listen 443 ssl http2; # 开启 HTTP/2
    server_name your.domain.com;

    # SSL 证书配置...
    ssl_certificate /path/to/your/cert.pem;
    ssl_certificate_key /path/to/your/key.pem;
}
</code></pre>
</li>
</ul>
</li>
</ol>
<h4 id="问题-5了解-nginx-的-location-匹配规则吗优先级是怎样的"><a href="#问题-5了解-nginx-的-location-匹配规则吗优先级是怎样的">问题 5：了解 Nginx 的 <code>location</code> 匹配规则吗？优先级是怎样的？</a></h4>
<p><strong>面试官意图：</strong> 考察你对 Nginx 配置核心逻辑的掌握程度。错误的 <code>location</code> 顺序可能导致配置不生效或被意外覆盖，这是排查问题的基础。</p>
<p><strong>回答思路：</strong></p>
<ol>
<li>
<p><strong>列出所有匹配规则：</strong></p>
<ul>
<li>
<p><code>=</code>：精确匹配。路径必须与 <code>location</code> 后的字符串完全相同。</p>
</li>
<li>
<p><code>^~</code>：前缀匹配（非正则）。如果匹配成功，则停止搜索其他正则匹配。</p>
</li>
<li>
<p><code>~</code>：正则表达式匹配（区分大小写）。</p>
</li>
<li>
<p><code>~*</code>：正则表达式匹配（不区分大小写）。</p>
</li>
<li>
<p><code>/</code>：通用前缀匹配。如果其他匹配都未成功，则会选择这个。</p>
</li>
</ul>
</li>
<li>
<p>解释优先级顺序（非常重要）：</p>
<p>Nginx 的 location 匹配遵循以下规则：</p>
<ul>
<li>
<p><strong>第一步：</strong> 检查所有<strong>精确匹配 (<code>=</code>)</strong>。如果找到，立即停止搜索，并使用该 <code>location</code>。</p>
</li>
<li>
<p><strong>第二步：</strong> 如果没有精确匹配，则检查所有<strong>前缀匹配 (<code>^~</code> 和普通前缀)</strong>。Nginx 会找到<strong>最长</strong>的那个前缀匹配项并<strong>记录</strong>下来。</p>
</li>
<li>
<p><strong>第三步：</strong> 如果被记录的最长前缀匹配是 <code>^~</code> 类型，那么就直接使用它，停止搜索。</p>
</li>
<li>
<p><strong>第四步：</strong> 如果被记录的最长前缀匹配是普通类型，Nginx 会<strong>继续</strong>检查<strong>正则表达式匹配 (<code>~</code> 和 <code>~*</code>)</strong>。它会按照配置文件中的<strong>书写顺序</strong>，从上到下进行检查。一旦找到第一个匹配的正则表达式，就立即停止搜索，并使用该 <code>location</code>。</p>
</li>
<li>
<p><strong>第五步：</strong> 如果没有任何一个正则表达式匹配成功，那么就使用<strong>第二步中记录的最长前缀匹配</strong>。</p>
</li>
</ul>
</li>
<li>
<p><strong>举一个综合例子：</strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx">location = / {
  # 规则 A
}
location / {
  # 规则 B
}
location /documents/ {
  # 规则 C
}
location ^~ /images/ {
  # 规则 D
}
location ~* \.(gif|jpg|jpeg)$ {
  # 规则 E
}
</code></pre>
<ul>
<li>
<p>访问 <code>/</code> -> 匹配 A</p>
</li>
<li>
<p>访问 <code>/index.html</code> -> 匹配 B（最长前缀 /，没有正则匹配）</p>
</li>
<li>
<p>访问 <code>/documents/about.html</code> -> 匹配 C（最长前缀 /documents/，没有正则匹配）</p>
</li>
<li>
<p>访问 <code>/images/logo.png</code> -> 匹配 D（<code>^~</code> 优先级高于正则，即使下面有正则能匹配 <code>.png</code>）</p>
</li>
<li>
<p>访问 <code>/static/avatar.gif</code> -> 匹配 E（最长前缀是 B，但继续搜索正则，E 匹配成功）</p>
</li>
</ul>
</li>
</ol>
<p><strong>最佳实践建议：</strong> 将不希望被正则覆盖的、确定的前缀路径用 <code>^~</code>，将需要精确匹配的用 <code>=</code>，这样可以使配置更加清晰和可控。</p>

<h3 id="第四层进阶与架构展示你的潜力与视野"><a href="#第四层进阶与架构展示你的潜力与视野">第四层：进阶与架构（展示你的潜力与视野）</a></h3>
<p>这类问题通常出现在资深岗位或技术面终面，用于判断你的技术深度和架构设计能力。</p>
<h4 id="问题-6除了反向代理你还知道-nginx-可以做什么吗比如负载均衡能简单配置一下吗"><a href="#问题-6除了反向代理你还知道-nginx-可以做什么吗比如负载均衡能简单配置一下吗">问题 6：除了反向代理，你还知道 Nginx 可以做什么吗？比如负载均衡，能简单配置一下吗？</a></h4>
<p><strong>面试官意图：</strong> 考察你的知识边界是否超出了纯粹的前端领域，是否具备对服务可用性的基本认知。</p>
<p><strong>回答思路：</strong></p>
<ol>
<li>
<p><strong>点明场景：</strong> 当单个应用服务器无法承受所有流量压力，或者为了保证服务的高可用（一台服务器挂了，其他还能顶上），就需要部署多台服务器。Nginx 的负载均衡功能就是将进来的请求分发给这些服务器。</p>
</li>
<li>
<p><strong>核心配置模块：<code>upstream</code></strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx"># 定义一个后端服务器集群
upstream my_backend_servers {
    # 可以设置权重，weight 越大，分配到的请求越多
    server 192.168.1.100:8080 weight=3;
    server 192.168.1.101:8080;
    server 192.168.1.102:8080 backup; # 标记为备份服务器
}

server {
    listen 80;
    server_name your.domain.com;

    location / {
        # 将请求转发到上面定义的服务器集群
        proxy_pass http://my_backend_servers;
    }
}
</code></pre>
</li>
<li>
<p><strong>介绍常见的负载均衡策略：</strong></p>
<ul>
<li>
<p><strong>轮询 (Round Robin)：</strong> 默认策略。请求按时间顺序逐一分配到不同的服务器，如果服务器 down 掉，能自动剔除。</p>
</li>
<li>
<p><strong>权重 (Weight)：</strong> <code>weight=number;</code>。指定轮询几率，<code>weight</code> 越高，分配到的请求就越多。适用于服务器性能不均衡的情况。</p>
</li>
<li>
<p><strong>IP Hash (<code>ip_hash;</code>)：</strong> 每个请求按访问 IP 的 hash 结果分配，这样每个访客固定访问一个后端服务器。可以解决 session 共享问题。</p>
</li>
<li>
<p><strong>最少连接 (<code>least_conn;</code>)：</strong> 请求会被转发到当前连接数最少的服务器。</p>
</li>
</ul>
</li>
</ol>
<h4 id="问题-7如何使用-nginx-配置-https你知道它的原理吗"><a href="#问题-7如何使用-nginx-配置-https你知道它的原理吗">问题 7：如何使用 Nginx 配置 HTTPS？你知道它的原理吗？</a></h4>
<p><strong>面试官意图：</strong> 考察安全意识和实践能力。在今天，HTTPS 几乎是网站标配。</p>
<p><strong>回答思路：</strong></p>
<ol>
<li>
<p><strong>配置步骤：</strong></p>
<ul>
<li>
<p><strong>获取证书：</strong> 首先需要有 SSL/TLS 证书。可以从商业机构购买，也可以使用免费的 Let's Encrypt。通常你会得到一个证书文件（如 <code>.pem</code> 或 <code>.crt</code>）和一个私钥文件（<code>.key</code>）。</p>
</li>
<li>
<p><strong>修改 Nginx 配置：</strong></p>
<p>Nginx</p>
<pre><code class="hljs language-nginx">server {
    listen 443 ssl http2; # 监听 443 端口，并启用 ssl 和 http2
    server_name your.domain.com;

    # 证书文件路径
    ssl_certificate      /etc/nginx/certs/your.domain.com.pem;
    # 私钥文件路径
    ssl_certificate_key  /etc/nginx/certs/your.domain.com.key;

    # 其他 SSL/TLS 安全配置 (可选但推荐)
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers 'TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384';
    ssl_prefer_server_ciphers on;

    # ... 其他 location 等配置
}

# 推荐配置：将所有 HTTP 请求重定向到 HTTPS
server {
    listen 80;
    server_name your.domain.com;
    return 301 https://$host$request_uri;
}
</code></pre>
</li>
</ul>
</li>
<li>
<p>简述原理 (TLS 握手过程)：</p>
<p>不需要说得像密码学专家，但要能讲清楚大致流程，体现你知其所以然。</p>
<ul>
<li>
<p><strong>客户端请求：</strong> 浏览器向服务器发送请求，带上自己支持的加密套件和随机数 C1。</p>
</li>
<li>
<p><strong>服务器响应：</strong> 服务器选择一个加密套件，返回自己的证书（包含公钥）和一个随机数 S1。</p>
</li>
<li>
<p><strong>客户端验证与密钥生成：</strong></p>
<ul>
<li>
<p>浏览器验证证书的有效性（是否由受信任的 CA 签发、是否过期、域名是否匹配）。</p>
</li>
<li>
<p>验证通过后，浏览器再生成一个随机数 P1（预主密钥），并用服务器的<strong>公钥</strong>加密它，发送给服务器。</p>
</li>
</ul>
</li>
<li>
<p><strong>服务器解密与会话密钥生成：</strong></p>
<ul>
<li>
<p>服务器用自己的<strong>私钥</strong>解密，得到预主密钥 P1。</p>
</li>
<li>
<p>现在，客户端和服务器都有了 C1, S1, P1 这三个随机数，双方用相同的算法，各自生成一个最终的<strong>会话密钥</strong>。</p>
</li>
</ul>
</li>
<li>
<p><strong>加密通信：</strong> 之后的所有通信，都使用这个<strong>对称</strong>的会话密钥进行加密和解密。</p>
</li>
</ul>
</li>
</ol>
<p><strong>总结：</strong> HTTPS 的核心就是通过<strong>非对称加密</strong>（公钥/私钥）安全地交换一个<strong>对称加密</strong>的密钥，之后再用这个对称密钥高效地进行通信。</p>
