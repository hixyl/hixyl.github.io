4:T2cc4,<p>对于初始渲染数据的获取，放在哪个生命周期是一个非常核心且重要的问题。答案根据你使用的组件类型（函数组件或类组件）而有所不同。</p>
<p><strong>总的来说，最佳实践是放在组件“挂载完成”的那个时间点。</strong></p>
<hr>
<h3 id="最佳选择-现代-react函数组件中的-useeffect"><a href="#最佳选择-现代-react函数组件中的-useeffect">最佳选择 (现代 React)：函数组件中的 <code>useEffect</code></a></h3>
<p>在现代的 React 开发中，我们首选函数组件和 Hooks。对于初始数据获取，你应该使用 <code>useEffect</code> Hook，并给它一个空的依赖数组 <code>[]</code>。</p>
<ul>
<li>
<p><strong>生命周期：</strong> <code>useEffect(() => { ... }, [])</code></p>
</li>
<li>
<p><strong>何时执行：</strong> 它会在组件的首次渲染（挂载）完成后执行，且<strong>只会执行一次</strong>。</p>
</li>
</ul>
<p><strong>为什么是这里？</strong></p>
<ol>
<li>
<p><strong>不阻塞渲染：</strong> 组件会先完成它的首次渲染（例如，显示一个 "Loading..." 的界面），然后 <code>useEffect</code> 里的代码才会执行。这保证了用户能立刻看到页面内容，而不是等待数据返回时看到一个白屏。</p>
</li>
<li>
<p><strong>副作用隔离：</strong> 数据获取是一种“副作用”（Side Effect），<code>useEffect</code> 的设计初衷就是用来处理这些副作用。</p>
</li>
<li>
<p><strong>防止重复获取：</strong> 空的依赖数组 <code>[]</code> 告诉 React，这个 effect 不依赖于任何 props 或 state，因此它只需要在组件“出生”时运行一次，之后无论组件如何更新，都不会再次运行。</p>
</li>
</ol>

<h3 id="传统选择类组件中的-componentdidmount"><a href="#传统选择类组件中的-componentdidmount">传统选择：类组件中的 <code>componentDidMount</code></a></h3>
<p>如果你在使用类组件（例如维护老项目），那么获取初始数据的正确位置是 <code>componentDidMount</code>。</p>
<ul>
<li>
<p><strong>生命周期：</strong> <code>componentDidMount()</code></p>
</li>
<li>
<p><strong>何时执行：</strong> 这个方法在组件实例被创建并插入到 DOM 后立即执行。它也只会在组件的整个生命周期中执行一次。</p>
</li>
</ul>
<p>它的理念和 <code>useEffect</code> 是一样的：在组件已经呈现在屏幕上之后，再开始进行数据请求等异步操作。</p>

<h3 id="绝对不能放数据请求的地方"><a href="#绝对不能放数据请求的地方">绝对不能放数据请求的地方</a></h3>
<ul>
<li>
<p><strong><code>render()</code> 方法 (类组件) 或函数组件的顶层作用域：</strong></p>
<ul>
<li><code>render</code> 必须是一个纯函数，只负责根据 <code>props</code> 和 <code>state</code> 返回 UI。在这里请求数据会导致每次 <code>state</code> 更新（比如 <code>setState({ loading: false })</code>）时都重新触发 <code>render</code>，从而再次发起请求，造成无限循环。</li>
</ul>
</li>
<li>
<p><strong><code>constructor()</code> (类组件)：</strong></p>
<ul>
<li>构造函数应该只用来初始化 <code>state</code> 和绑定方法。在这里发起异步请求是一种反模式，因为在数据返回并调用 <code>setState</code> 之前，组件可能已经完成了挂载，这会使逻辑变得混乱且难以维护。</li>
</ul>
</li>
</ul>
