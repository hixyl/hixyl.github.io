<h1 id="css-优先级算法如何计算"><a href="#css-优先级算法如何计算">CSS 优先级算法如何计算？</a></h1>
<ul>
<li>优先级就近原则，同权重情况下样式定义最近者为准;</li>
<li>载入样式以最后载入的定位为准;</li>
</ul>
<p>优先级为:</p>
<ul>
<li>同权重: 内联样式表（标签内部）> 嵌入样式表（当前文件中）> 外部样式表（外部文件中）。</li>
<li>!important > id > class > tag</li>
<li>!important 比 内联优先级高</li>
</ul>
<h1 id="-min-widthmax-widthwidth-的包含优先级关系"><a href="#-min-widthmax-widthwidth-的包含优先级关系">🔥 <strong>min-width、max-width、width 的包含（优先级）关系</strong></a></h1>
<p><strong>属性的含义：</strong></p>
<ul>
<li><code>min-width</code> 限制元素的最小宽度</li>
<li><code>max-width</code> 限制元素的最大宽度</li>
<li><code>width</code> 元素的宽度</li>
</ul>
<p><strong>三者之间的优先级：</strong></p>
<p><code>min-width</code> 和 <code>max-width</code> 的优先级都高于 <code>width</code>。即使 <code>width</code> 后面加上 <code>!important</code>。</p>
<ul>
<li>当浏览器缩小导致元素宽度小于 <code>min-width</code> 时，元素的 <code>width</code> 就会被 <code>min-width</code> 的值取代，浏览器出现滚动条来容纳元素。</li>
<li>当浏览器放大导致元素的宽度大于 <code>max-width</code> 时，元素的 <code>width</code> 就会被 <code>max-width</code> 值取代。</li>
<li>当 <code>min-width</code> 值大于 <code>max-width</code> 时，则以 <code>min-width</code> 值为准。</li>
</ul>
<p><strong>所以三者优先级排序： <code>min-width > max-width > width</code></strong></p>
<p>当 <code>min-width > width</code> 时，以 <code>min-width</code> 为主</p>

<h1 id="bfcblock-formating-context"><a href="#bfcblock-formating-context">BFC（Block Formating Context）</a></h1>
<ul>
<li>理解
<ul>
<li>拥有一些特性，可以利用其帮助我们布局</li>
</ul>
</li>
<li><strong>触发条件</strong>
<ul>
<li>根元素，即 HTML 元素</li>
<li>float: left | right</li>
<li>position: absolute | fixed</li>
<li>display: inline-block | table-cell | flex | inline-flex</li>
<li>overflow: hidden | auto | scroll</li>
</ul>
</li>
<li>解决问题
<ul>
<li>清除浮动：
<ul>
<li>子浮动，父 <code>overflow: hidden</code>（缺点：阴影和下拉菜单）</li>
</ul>
</li>
<li>div 垂直方向 margin 上下合并：
<ul>
<li>其中一个包 div ，设置 <code>overflow：hidden</code></li>
</ul>
</li>
<li>div 垂直方向 margin 内外合并：
<ul>
<li>父容器 1px 透明上边框</li>
<li>父容器 <code>overflow: hidden</code></li>
</ul>
</li>
<li>右侧 div 自适应：左边浮动，右边设置 <code>overflow：hidden</code></li>
</ul>
</li>
</ul>
<h1 id="-请解释一下为什么需要清除浮动清除浮动的方式"><a href="#-请解释一下为什么需要清除浮动清除浮动的方式">🔥 请解释一下为什么需要清除浮动？清除浮动的方式</a></h1>
<p><a href="https://www.notion.so/1e2d5abe78f84152b6c75e5709fd4a24?pvs=21">清除浮动</a></p>
<ul>
<li>原本作用
<ul>
<li>浮动本质是用来做一些文字混排效果的，但是被我们拿来做布局用，则会有很多的问题出现。</li>
</ul>
</li>
<li>带来的影响
<ul>
<li>由于浮动元素<strong>不再占用原文档流</strong>的位置，所以它会对后面的元素<strong>排版产生影响</strong>（div忽视它的存在，但文字环绕），为了解决这些问题，此时就需要在该元素中清除浮动。准确地说，并不是清除浮动，而是<strong>清除浮动后造成的影响</strong></li>
</ul>
</li>
<li>本质
<ul>
<li>清除浮动主要为了解决父级元素因为子级浮动引起内部高度为 0 的问题。</li>
</ul>
</li>
<li>方法
<ul>
<li>
<p>额外标签 <code>clear: both</code></p>
</li>
<li>
<p>BFC父元素 <code>overflow：hidden</code></p>
</li>
<li>
<p>伪元素</p>
<pre><code class="hljs language-css"><span class="hljs-selector-class">.clearfix</span><span class="hljs-selector-pseudo">::after</span> {
  <span class="hljs-attribute">content</span>: <span class="hljs-string">""</span>;
  <span class="hljs-attribute">display</span>: block;
  <span class="hljs-attribute">height</span>: <span class="hljs-number">0</span>;
  <span class="hljs-attribute">clear</span>: both;
  <span class="hljs-attribute">visibility</span>: hidden;
}
</code></pre>
</li>
</ul>
</li>
</ul>

<h1 id="-你对-line-height-是如何理解的"><a href="#-你对-line-height-是如何理解的">🔥 你对 line-height 是如何理解的？</a></h1>
<p>行高是指一行文字的高度，具体说是<strong>两行文字间基线的距离</strong>。CSS中起高度作用的是 height 和 line-height，没有定义 height 属性，最终其表现作用一定是 line-height 。</p>
<h1 id="-display-nonevisibility-hidden-和-opacity-0-的区别"><a href="#-display-nonevisibility-hidden-和-opacity-0-的区别">🔥 display: none、visibility: hidden 和 opacity: 0 的区别？</a></h1>
<ul>
<li><code>display: none</code> （不占空间，不能点击）（回流+重绘）</li>
<li><code>visibility: hidden</code> （<strong>占据空间</strong>，不可点击）（重绘）</li>
<li><code>opacity: 0</code>（<strong>占据空间</strong>，可以点击）（重建图层，性能较高）</li>

<h1 id="-css-动画与-js-动画哪个性能更好"><a href="#-css-动画与-js-动画哪个性能更好">🔥 CSS 动画与 JS 动画哪个性能更好</a></h1>
<p>事实上，大多数场景下，基于 CSS 的动画几乎是跟 JavaScript 动画表现一致——至少在 FireFox 上是如此。一些基于 Javascript 的动画库，像 <a href="https://greensock.com/gsap/">GSAP</a> 和 <a href="http://velocityjs.org/">Velocity.JS</a>，甚至声称他们在性能上可以做得比<a href="https://css-tricks.com/myth-busting-css-animations-vs-javascript/">原生 CSS transition/animation</a> 更好。这是可能的，因为在重绘事件发生之前，CSS transition 和 animation 在主的 UI 线程仅仅是重新采集元素的样式，这跟通过 <code>requestAnimationFrame()</code> 回调获取重新采集元素样式是一样的，也是在下一次重绘之前触发。假如二者都是在主 UI 线程创建的动画，那它们在性能方面没有差异。</p>
<p>但我们仍然认为 CSS 动画是更好的选择。为什么？关键是只要动画涉及的属性不引起 reflow（重新布局）（参考 <a href="https://csstriggers.com/">CSS trigger</a> 获得更多信息），我们可以把采样操作移出主线程。最常见的属性是 CSS transform。如果一个元素被提升为一个 <a href="https://wiki.mozilla.org/Gecko:Overview#Graphics">layer</a>，transform 属性动画就可以在 GPU 中进行。这意味着更好地性能，特别是在移动设备上。在 <a href="https://wiki.mozilla.org/Platform/GFX/OffMainThreadCompositing">OffMainThreadCompositing</a> 上寻找更多细节。</p>
<p>来源：</p>
<p><a href="https://developer.mozilla.org/zh-CN/docs/Web/Performance/CSS_JavaScript_animation_performance">CSS 动画与 JavaScript 动画的性能 - Web 性能 | MDN</a></p>
<ul>
<li>CSS3 的动画：
<ol>
<li>在性能上会稍微好一些，浏览器会对 CSS3 的动画做一些优化（比如专门新建一个图层用来跑动画）</li>
<li>代码相对简单</li>
<li>在动画控制上不够灵活</li>
<li>兼容性不好</li>
<li>部分动画功能无法实现（如滚动动画，视差滚动等）</li>
</ol>
</li>
<li>JavaScript 的动画：
<ol>
<li>正好弥补了 css 缺点，控制能力很强，可以单帧的控制、变换，同时写得好完全可以兼容 IE6</li>
<li>功能强大</li>
<li>如果动画比较复杂，在使用 <strong><code>requestAnimationFrame</code></strong> 时，回调函数中执行了大量的计算或操作，导致了每帧的执行时间过长。这样就会使得浏览器无法在每一帧中及时完成绘制，从而出现卡顿的情况。</li>
</ol>
</li>
<li>总结：
<ul>
<li>对于一些复杂控制的动画，使用 javascript 会比较好。而在实现一些小的交互动效的时候，可以多考虑 CSS</li>
</ul>
</li>
</ul>
