19:T3f21,<p>好的，我们来深入探讨一下前端开发中的策略模式。</p>
<p>我会从它是什么、为什么需要它、以及如何在实际工作中运用它这几个方面，结合具体的代码示例来阐述我的理解。</p>
<h3 id="核心理解将做什么与怎么做分离开"><a href="#核心理解将做什么与怎么做分离开">核心理解：将“做什么”与“怎么做”分离开</a></h3>
<p>想象一个场景：你在一个电商网站结算，可以选择多种支付方式：支付宝、微信支付、信用卡支付。</p>
<ul>
<li>
<p><strong>“做什么”</strong>：是固定的，就是“发起支付”。</p>
</li>
<li>
<p><strong>“怎么做”</strong>：是变化的，用支付宝、用微信、还是用信用卡，每种方式的实现细节都完全不同。</p>
</li>
</ul>
<p>策略模式的核心思想就是，把这些变化的“怎么做”（具体的算法或行为），从“做什么”（执行这个行为的上下文）中抽离出来，把每一个“怎么做”都封装成一个独立的、可以相互替换的“策略”。</p>
<p>最终，执行者（Context）不需要关心内部具体是如何实现的，只需要在需要的时候，告诉它使用哪个“策略”即可。</p>
<hr>
<h3 id="为什么要用策略模式告别庞大的-if-else--switch"><a href="#为什么要用策略模式告别庞大的-if-else--switch">为什么要用策略模式？告别庞大的 <code>if-else</code> / <code>switch</code></a></h3>
<p>在没有使用策略模式之前，我们可能会写出这样的代码来实现支付功能：</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">function</span> <span class="hljs-title function_">processPayment</span>(<span class="hljs-params">type, amount</span>) {
  <span class="hljs-keyword">if</span> (type === <span class="hljs-string">'alipay'</span>) {
    <span class="hljs-comment">// 调用支付宝的 SDK，处理参数，跳转...</span>
    <span class="hljs-variable language_">console</span>.<span class="hljs-title function_">log</span>(<span class="hljs-string">`使用支付宝支付 <span class="hljs-subst">${amount}</span> 元`</span>);
  } <span class="hljs-keyword">else</span> <span class="hljs-keyword">if</span> (type === <span class="hljs-string">'wechat'</span>) {
    <span class="hljs-comment">// 生成微信支付二维码，轮询支付状态...</span>
    <span class="hljs-variable language_">console</span>.<span class="hljs-title function_">log</span>(<span class="hljs-string">`使用微信支付 <span class="hljs-subst">${amount}</span> 元`</span>);
  } <span class="hljs-keyword">else</span> <span class="hljs-keyword">if</span> (type === <span class="hljs-string">'creditcard'</span>) {
    <span class="hljs-comment">// 连接银行网关，加密卡号信息...</span>
    <span class="hljs-variable language_">console</span>.<span class="hljs-title function_">log</span>(<span class="hljs-string">`使用信用卡支付 <span class="hljs-subst">${amount}</span> 元`</span>);
  } <span class="hljs-keyword">else</span> {
    <span class="hljs-comment">// 默认或错误处理</span>

  }
}

<span class="hljs-title function_">processPayment</span>(<span class="hljs-string">'alipay'</span>, <span class="hljs-number">100</span>);
</code></pre>
<p>这段代码有几个显而易见的问题：</p>
<ol>
<li>
<p><strong>违反开放/封闭原则</strong>：如果未来要增加一种新的支付方式，比如“数字货币支付”，你必须修改 <code>processPayment</code> 函数的内部逻辑，增加一个 <code>else if</code> 分支。这使得函数越来越臃肿，难以维护。</p>
</li>
<li>
<p><strong>职责耦合</strong>：<code>processPayment</code> 函数承担了太多的职责。它既要知道“发起支付”这个行为，又要知道所有支付方式的具体实现细节。</p>
</li>
<li>
<p><strong>复用性差</strong>：如果另一个业务场景也需要用到支付宝支付的逻辑，你可能需要复制代码，或者将这个巨大的函数到处引用。</p>
</li>
</ol>
<p>策略模式正是为了解决这些问题而生的。</p>
<hr>
<h3 id="如何在前端中应用策略模式以表单验证为例"><a href="#如何在前端中应用策略模式以表单验证为例">如何在前端中应用策略模式？以表单验证为例</a></h3>
<p>表单验证是前端最常见的场景之一，也是策略模式的绝佳应用场景。一个输入框可能有多种验证规则，比如“不能为空”、“必须是手机号”、“长度不能少于8位”等等。</p>
<h4 id="第1步定义一组策略strategies"><a href="#第1步定义一组策略strategies">第1步：定义一组策略（Strategies）</a></h4>
<p>我们将每一种验证规则都封装成一个独立的函数。这些函数遵循同样的接口约定（例如，都接收 <code>value</code> 和 <code>errorMsg</code> 两个参数，返回错误信息或者 <code>undefined</code>）。</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-comment">// 策略对象，用于存放所有验证规则</span>
<span class="hljs-keyword">const</span> strategies = {
  <span class="hljs-attr">isNonEmpty</span>: <span class="hljs-keyword">function</span>(<span class="hljs-params">value, errorMsg</span>) {
    <span class="hljs-keyword">if</span> (value === <span class="hljs-string">''</span> || value === <span class="hljs-literal">undefined</span> || value === <span class="hljs-literal">null</span>) {
      <span class="hljs-keyword">return</span> errorMsg;
    }
  },
  <span class="hljs-attr">minLength</span>: <span class="hljs-keyword">function</span>(<span class="hljs-params">value, length, errorMsg</span>) {
    <span class="hljs-keyword">if</span> (value.<span class="hljs-property">length</span> &#x3C; length) {
      <span class="hljs-keyword">return</span> errorMsg;
    }
  },
  <span class="hljs-attr">isMobile</span>: <span class="hljs-keyword">function</span>(<span class="hljs-params">value, errorMsg</span>) {
    <span class="hljs-comment">// 一个简单的手机号正则</span>
    <span class="hljs-keyword">if</span> (!<span class="hljs-regexp">/(^1[3|4|5|6|7|8|9][0-9]{9}$)/</span>.<span class="hljs-title function_">test</span>(value)) {
      <span class="hljs-keyword">return</span> errorMsg;
    }
  }
};
</code></pre>
<p><strong>这里的 <code>strategies</code> 对象就是我们的策略库。每一个键值对都是一个独立的策略。</strong></p>
<h4 id="第2步创建上下文context"><a href="#第2步创建上下文context">第2步：创建上下文（Context）</a></h4>
<p>上下文是使用这些策略的“执行者”。在表单验证的例子中，我们可以创建一个 <code>Validator</code> 类来扮演这个角色。它负责接收和管理需要执行的验证规则，并最终执行它们。</p>
<p>JavaScript</p>
<pre><code class="hljs language-javascript"><span class="hljs-keyword">class</span> <span class="hljs-title class_">Validator</span> {
  <span class="hljs-title function_">constructor</span>(<span class="hljs-params"></span>) {
    <span class="hljs-variable language_">this</span>.<span class="hljs-property">cache</span> = []; <span class="hljs-comment">// 用于存储要执行的验证规则</span>
  }

  <span class="hljs-comment">// 添加规则</span>
  <span class="hljs-title function_">add</span>(<span class="hljs-params">dom, rules</span>) {
    <span class="hljs-keyword">for</span> (<span class="hljs-keyword">let</span> rule <span class="hljs-keyword">of</span> rules) {
      <span class="hljs-comment">// 分离策略名称和参数</span>
      <span class="hljs-keyword">let</span> strategyArr = rule.<span class="hljs-property">strategy</span>.<span class="hljs-title function_">split</span>(<span class="hljs-string">':'</span>);
      <span class="hljs-keyword">let</span> errorMsg = rule.<span class="hljs-property">errorMsg</span>;

      <span class="hljs-variable language_">this</span>.<span class="hljs-property">cache</span>.<span class="hljs-title function_">push</span>(<span class="hljs-function">() =></span> {
        <span class="hljs-keyword">let</span> strategy = strategyArr[<span class="hljs-number">0</span>]; <span class="hljs-comment">// e.g., 'minLength'</span>
        strategyArr.<span class="hljs-title function_">shift</span>(); <span class="hljs-comment">// 移除策略名，剩下参数</span>
        <span class="hljs-keyword">let</span> args = [dom.<span class="hljs-property">value</span>, ...strategyArr, errorMsg]; <span class="hljs-comment">// 组合参数</span>
        
        <span class="hljs-comment">// 调用策略函数</span>
        <span class="hljs-keyword">return</span> strategies[strategy].<span class="hljs-title function_">apply</span>(dom, args);
      });
    }
  }

  <span class="hljs-comment">// 开始验证</span>
  <span class="hljs-title function_">start</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">for</span> (<span class="hljs-keyword">let</span> validatorFunc <span class="hljs-keyword">of</span> <span class="hljs-variable language_">this</span>.<span class="hljs-property">cache</span>) {
      <span class="hljs-keyword">const</span> errorMsg = <span class="hljs-title function_">validatorFunc</span>(); <span class="hljs-comment">// 执行验证函数</span>
      <span class="hljs-keyword">if</span> (errorMsg) {
        <span class="hljs-keyword">return</span> errorMsg; <span class="hljs-comment">// 如果有错误信息，立即返回</span>
      }
    }
  }
}
</code></pre>
<p><strong>这个 <code>Validator</code> 类就是我们的上下文。</strong> 它内部并不包含任何具体的验证逻辑。它只做三件事：</p>
<ol>
<li>
<p>提供 <code>add</code> 方法，让外部告诉它要对哪个输入框（<code>dom</code>）使用哪些策略（<code>rules</code>）。</p>
</li>
<li>
<p>将这些待执行的策略（封装成函数）存放在 <code>cache</code> 数组里。</p>
</li>
<li>
<p>提供 <code>start</code> 方法，遍历 <code>cache</code>，依次调用策略，并返回结果。</p>
</li>
</ol>

<p>现在，我们可以非常灵活地使用它了。</p>

<h3 id="策略模式的优势总结"><a href="#策略模式的优势总结">策略模式的优势总结</a></h3>
<ol>
<li>
<p><strong>高度解耦，易于维护</strong>：验证逻辑（策略）和业务逻辑（上下文）完全分离。你可以随意修改、增加、删除任何一个验证规则，而完全不需要动 <code>Validator</code> 类的代码。这完美符合<strong>开放/封闭原则</strong>。</p>
</li>
<li>
<p><strong>策略复用</strong>：<code>isNonEmpty</code>、<code>isMobile</code> 这些策略都是纯函数，可以在项目的任何地方被复用，甚至可以打包成一个独立的工具库。</p>
</li>
<li>
<p><strong>可测试性强</strong>：每个策略都是一个独立的小单元，非常容易进行单元测试。</p>
</li>
<li>
<p><strong>代码更清晰</strong>：避免了冗长的 <code>if-else</code> 结构，让代码的意图更加明显和优雅。</p>
</li>
</ol>
<h3 id="我的看法与最佳选择"><a href="#我的看法与最佳选择">我的看法与最佳选择</a></h3>
<p>策略模式并非银弹，它通过增加一些结构来换取灵活性和可维护性。</p>
<ul>
<li>
<p><strong>对于简单场景</strong>：如果你的逻辑只有两种固定的情况，比如 <code>if (isVip) { ... } else { ... }</code>，那么直接使用 <code>if-else</code> 反而更简单直接，强行使用策略模式会显得过度设计。</p>
</li>
<li>
<p><strong>对于复杂、多变、可扩展的场景</strong>：当你有3个或更多个可选的行为，并且这些行为未来还可能增加或改变时，策略模式就是你的最佳选择。前端的表单验证、动画效果选择、数据上报方式、不同主题的渲染逻辑等，都非常适合使用策略模式来构建。</p>
</li>
</ul>
