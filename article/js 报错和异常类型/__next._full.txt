18:T1820,<p>在 JavaScript 中，报错（Error）不仅仅是程序的“罢工”，它是一套完整的对象体系。当代码运行出现问题时，JavaScript 引擎会根据问题的本质，抛出对应类型的错误对象。</p>
<p>这些错误都继承自基类 <strong><code>Error</code></strong>。理解它们能让你在调试时迅速定位是逻辑出了问题，还是语法或是内存超限。</p>
<hr>
<h3 id="常见的内置错误类型"><a href="#常见的内置错误类型">常见的内置错误类型</a></h3>
<table>
<thead>
<tr>
<th><strong>错误类型</strong></th>
<th><strong>触发场景</strong></th>
<th><strong>典型例子</strong></th>
</tr>
</thead>
<tbody>
<tr>
<td><strong><code>SyntaxError</code></strong></td>
<td><strong>语法错误</strong>。代码不符合 JS 规范，解析阶段就报错。</td>
<td>变量名命名非法、括号没闭合。</td>
</tr>
<tr>
<td><strong><code>ReferenceError</code></strong></td>
<td><strong>引用错误</strong>。尝试访问一个不存在（未声明）的变量。</td>
<td>使用未定义的变量 <code>console.log(a)</code>。</td>
</tr>
<tr>
<td><strong><code>TypeError</code></strong></td>
<td><strong>类型错误</strong>。值的类型不是预期类型。</td>
<td>对 <code>null</code> 读取属性、调用一个非函数对象。</td>
</tr>
<tr>
<td><strong><code>RangeError</code></strong></td>
<td><strong>范围错误</strong>。数值超出了允许的范围。</td>
<td>数组长度为负数、递归过深导致栈溢出。</td>
</tr>
<tr>
<td><strong><code>URIError</code></strong></td>
<td><strong>URI 错误</strong>。相关编码/解码函数使用不当。</td>
<td><code>decodeURI('%')</code>（百分号后缺少字符）。</td>
</tr>
<tr>
<td><strong><code>EvalError</code></strong></td>
<td><strong>Eval 错误</strong>。与 <code>eval()</code> 函数相关的错误。</td>
<td>在现代 JS 中已极少见，主要为兼容性存在。</td>
</tr>
<tr>
<td><strong><code>AggregateError</code></strong></td>
<td><strong>集合错误</strong>。当一个操作需要报告多个错误时使用。</td>
<td><code>Promise.any()</code> 中所有 Promise 都 rejected。</td>
</tr>
</tbody>
</table>
<hr>
<h3 id="深度解构为什么会报错"><a href="#深度解构为什么会报错">深度解构：为什么会报错？</a></h3>
<h4 id="1-syntaxerror编译器无法理解你"><a href="#1-syntaxerror编译器无法理解你">1. SyntaxError：编译器无法理解你</a></h4>
<p>这是唯一一个在代码<strong>执行前</strong>就会触发的错误。如果脚本中有语法错误，整段脚本都不会运行。</p>
<ul>
<li><strong>例子</strong>：<code>let 123var = 'error';</code>（变量名不能以数字开头）。</li>
</ul>
<h4 id="2-referenceerror作用域的迷失"><a href="#2-referenceerror作用域的迷失">2. ReferenceError：作用域的迷失</a></h4>
<p>当你试图读取一个在当前作用域及其父级作用域都找不到的变量名时，引擎会放弃搜索并抛出此异常。</p>
<ul>
<li><strong>注意</strong>：在 ES6 之后，如果你在 <code>let</code> 或 <code>const</code> 声明之前访问该变量，会进入“暂时性死区”，抛出的也是引用错误。</li>
</ul>
<h4 id="3-typeerror最常见的运行时杀手"><a href="#3-typeerror最常见的运行时杀手">3. TypeError：最常见的“运行时”杀手</a></h4>
<p>这通常意味着你的逻辑在某个环节断了。比如你以为某个 API 返回的是数组，结果它返回了 <code>undefined</code>，而你却尝试对它进行 <code>.map()</code> 操作。</p>
<ul>
<li><strong>现象</strong>：<code>Uncaught TypeError: Cannot read properties of undefined (reading 'xxx')</code>。</li>
</ul>
<h4 id="4-rangeerror触碰了物理或逻辑极限"><a href="#4-rangeerror触碰了物理或逻辑极限">4. RangeError：触碰了物理或逻辑极限</a></h4>
<p>最典型的场景是<strong>死循环递归</strong>。每一个函数调用都会占用一定的栈内存（Stack），当递归没有出口时，栈会被撑破。</p>
<ul>
<li><strong>公式表达</strong>：若递归深度 <span class="katex"><span class="katex-mathml"><math xmlns="http://www.w3.org/1998/Math/MathML"><semantics><mrow><mi>n</mi><mo>→</mo><mi mathvariant="normal">∞</mi></mrow><annotation encoding="application/x-tex">n \to \infty</annotation></semantics></math></span><span class="katex-html" aria-hidden="true"><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord mathnormal">n</span><span class="mspace" style="margin-right:0.2778em;"></span><span class="mrel">→</span><span class="mspace" style="margin-right:0.2778em;"></span></span><span class="base"><span class="strut" style="height:0.4306em;"></span><span class="mord">∞</span></span></span></span>，则必触发 <code>InternalError</code>（非标）或 <code>RangeError</code>。</li>
</ul>
<hr>
<h3 id="异常的处理与自定义"><a href="#异常的处理与自定义">异常的处理与自定义</a></h3>
<p>除了被动接受系统的报错，我们还可以主动“制造”异常来增强程序的健壮性。</p>
<ul>
<li>
<p><strong>抛出异常</strong>：你可以使用 <code>throw</code> 关键字抛出任何东西，但最佳实践是抛出一个 <code>Error</code> 实例：</p>
<p><code>throw new Error('自定义的错误信息');</code></p>
</li>
<li>
<p><strong>捕获异常</strong>：通过 <code>try...catch...finally</code> 结构。这能让程序在遇到局部错误时不至于崩溃，而是优雅地降级处理。</p>
</li>
</ul>
<h3 id="进阶思考异步中的报错"><a href="#进阶思考异步中的报错">进阶思考：异步中的报错</a></h3>
