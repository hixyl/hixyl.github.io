1a:T40ac,<h3 id="1-问题背景"><a href="#1-问题背景">1. 问题背景</a></h3>
<p>项目中有一个可复用的 React 组件 <code>YlCollapse</code>，它实现了一个点击标题可以展开/收起内容的折叠面板。其动画效果是通过 CSS <code>transition</code> 和 <code>max-height</code> 属性实现的。</p>
<p><strong>遇到的问题是：</strong> 当该组件包裹的内容（例如一个 antd <code>Table</code> 组件）在展开状态下，因用户操作（如切换分页）导致其自身高度突然增加时，<code>YlCollapse</code> 组件的容器没有随之适应新的高度，导致新增的内容被截断，无法完整显示。</p>
<h3 id="2-根源分析"><a href="#2-根源分析">2. 根源分析</a></h3>
<p>问题的核心在于 <code>ResizeObserver</code> 的观察目标与 <code>max-height</code> 动画机制之间产生了逻辑冲突。</p>
<ol>
<li>
<p><strong>动画机制</strong>：组件利用 <code>overflow: hidden</code> 和 <code>max-height</code> 从 <code>0px</code> 到 <code>content.scrollHeight</code> 的 <code>transition</code> 来实现平滑的展开动画。</p>
</li>
<li>
<p><strong><code>ResizeObserver</code> 的用法</strong>：原有代码使用 <code>ResizeObserver</code> 来侦测内容容器 <code>div.content</code> 本身的尺寸变化，期望在内容高度变化时自动更新 <code>max-height</code>。</p>
</li>
<li>
<p><strong>核心冲突点（死循环）</strong>：<code>ResizeObserver</code> 侦测的是元素<strong>最终渲染的盒子尺寸</strong>（<code>height</code>）。然而，<code>div.content</code> 的渲染高度已经被 <code>max-height</code> 属性“锁死”了。当内部的 <code>Table</code> 变高时，<code>div.content</code> 的 <code>scrollHeight</code>（可滚动高度）确实增加了，但它的 <code>height</code>（渲染高度）并未改变。因此，<code>ResizeObserver</code> 不会触发回调，<code>max-height</code> 的值也就无法得到更新。这是一个典型的“死循环”陷阱。</p>
</li>
<li>
<p><strong>次要问题</strong>：代码中使用了 <code>document.querySelector</code> 来获取 DOM 节点。在 React 中，这是一种指令式的反模式（anti-pattern），它破坏了组件的封装性，并且在复杂的 DOM 结构或多个组件实例共存时可能导致意想不到的 bug。</p>
</li>
</ol>
<h3 id="3-解决方案与思路"><a href="#3-解决方案与思路">3. 解决方案与思路</a></h3>
<p>我们采用**“职责分离”**的原则来重构组件逻辑，彻底解决此问题。</p>
<ol>
<li>
<p><strong>分离动画与测量</strong>：我们引入一个额外的内层 <code>div</code> 作为包裹层。</p>
<ul>
<li>
<p><strong>外层容器 (<code>div.content</code>)</strong>：<strong>只负责动画</strong>。它拥有 <code>max-height</code> 和 <code>overflow: hidden</code> 样式。</p>
</li>
<li>
<p><strong>内层包裹层 (<code>wrapper</code>)</strong>：<strong>只负责测量</strong>。它的高度不受任何限制，会随着子内容自由伸缩。</p>
</li>
</ul>
</li>
<li>
<p><strong>修正观察目标</strong>：将 <code>ResizeObserver</code> 的观察目标从外层容器改为这个不受约束的内层包裹层。这样，只要内部 <code>Table</code> 高度变化，内层的高度就会变化，<code>ResizeObserver</code> 就能被正确触发。</p>
</li>
<li>
<p><strong>拥抱 React Hooks</strong>：使用 <code>useRef</code> Hook 来替代 <code>document.querySelector</code>，以声明式、更安全的方式获取 DOM 节点的引用。</p>
</li>
<li>
<p><strong>优化渲染性能</strong>：在 <code>ResizeObserver</code> 的回调中，使用 <code>requestAnimationFrame</code> (rAF) 来包裹更新 <code>max-height</code> 的操作。这可以将 DOM 的更新操作与浏览器的渲染周期同步，避免因连续的“读/写”操作引发的“布局抖动”（Layout Thrashing），确保动画在内容频繁变化时依然保持流畅。</p>
</li>
</ol>

<h3 id="5-总结与最佳实践"><a href="#5-总结与最佳实践">5. 总结与最佳实践</a></h3>
<p>此次修复不仅解决了眼前的 bug，也为我们沉淀了关于动态内容与 CSS 动画交互的最佳实践。</p>
<ol>
<li>
<p><strong>掌握 <code>max-height</code> 动画模式</strong>：当使用 <code>max-height</code> 制作内容自适应的动画时，必须采用“外层动画，内层测量”的嵌套结构，这是解决此类问题的标准模式。</p>
</li>
<li>
<p><strong>坚持 React 的声明式思想</strong>：在组件内部，应始终优先使用 <code>useRef</code> 来操作和引用 DOM 元素，避免使用 <code>document.querySelector</code> 等指令式 API，以保证组件的健壮性和可维护性。</p>
</li>
<li>
<p><strong>关注渲染性能</strong>：对于涉及 DOM 尺寸读取和样式写入的交互，尤其是那些可能被频繁触发的场景（如 <code>ResizeObserver</code>, <code>onScroll</code>），应习惯性地使用 <code>requestAnimationFrame</code> 进行包裹，将性能优化作为编码习惯的一部分。</p>
</li>
<li>
<p><strong>保持组件封装</strong>：一个好的组件应该管理好自己内部的复杂性。此次修复将所有逻辑都封装在 <code>YlCollapse</code> 内部，对父组件完全透明，父组件无需关心其内部实现细节，这符合高内聚、低耦合的设计原则。</p>
</li>
