1c:T318b,<p>我们来系统且深入地剖析 <code>&#x3C;script></code> 标签，从 <code>defer</code> 属性开始，并延展到所有相关的属性。</p>
<p>这不仅仅是关于“功能是什么”，更是关于“为什么需要它”以及“在何种场景下做出最优选择”。</p>
<h3 id="defer-的核心作用延迟且有序的执行"><a href="#defer-的核心作用延迟且有序的执行"><code>defer</code> 的核心作用：延迟且有序的执行</a></h3>
<p><code>defer</code> 的全称是 "defer execution"。它的核心目的，是在不阻塞页面渲染的前提下，加载并执行 JavaScript 代码。</p>
<p>要理解 <code>defer</code>，首先要明白浏览器默认是如何处理 <code>&#x3C;script></code> 标签的。</p>
<h4 id="1-默认行为无任何属性"><a href="#1-默认行为无任何属性">1. 默认行为（无任何属性）</a></h4>
<p>当浏览器解析 HTML 文档时，从上到下逐行处理。如果遇到一个普通的 <code>&#x3C;script src="...">&#x3C;/script></code> 标签：</p>
<ol>
<li>
<p><strong>停止解析</strong>：浏览器会立即暂停解析后续的 HTML。</p>
</li>
<li>
<p><strong>下载脚本</strong>：它会发起请求去下载该脚本文件。</p>
</li>
<li>
<p><strong>执行脚本</strong>：下载完成后，立即执行这个脚本。</p>
</li>
<li>
<p><strong>继续解析</strong>：脚本执行完毕后，才继续解析剩余的 HTML。</p>
</li>
</ol>
<p>这个过程是<strong>阻塞式 (Blocking)</strong> 的。如果脚本很大或者网络很慢，用户就会长时间看到一个白屏，因为浏览器被卡住，无法渲染页面的其余部分。这带来了极差的用户体验。</p>
<p><em>(这是一个简化的示意图，红色代表阻塞)</em></p>
<h4 id="2-defer-的解决方案"><a href="#2-defer-的解决方案">2. <code>defer</code> 的解决方案</a></h4>
<p>为了解决上述问题，<code>defer</code> 属性应运而生。当你在 script 标签上加上 <code>defer</code> 时，行为会变成这样：</p>
<ol>
<li>
<p><strong>并行下载</strong>：浏览器看到 <code>&#x3C;script defer src="...">&#x3C;/script></code>，会立即开始<strong>异步下载</strong>脚本文件，但<strong>不会停止</strong>解析后续的 HTML。下载过程和 HTML 解析是并行的。</p>
</li>
<li>
<p><strong>延迟执行</strong>：脚本下载完成后并不会立即执行，而是会等待整个 HTML 文档都被解析完毕（即 <code>DOMContentLoaded</code> 事件触发之前）。</p>
</li>
<li>
<p><strong>有序执行</strong>：如果页面上有多个带 <code>defer</code> 的脚本，它们会严格按照在 HTML 中出现的顺序来执行。第一个 <code>defer</code> 脚本总是先于第二个 <code>defer</code> 脚本执行。</p>
</li>
</ol>
<p><strong><code>defer</code> 的核心优势：</strong></p>
<ul>
<li>
<p><strong>非阻塞渲染</strong>：下载过程不影响页面渲染，用户能更快地看到完整页面结构。</p>
</li>
<li>
<p><strong>DOM可用</strong>：执行时，整个 DOM 树已经构建完毕，脚本可以安全地访问页面上的任何元素。</p>
</li>
<li>
<p><strong>执行顺序保证</strong>：对于有依赖关系的脚本（例如，<code>jquery.js</code> 必须在 <code>main.js</code> 之前执行），<code>defer</code> 能够确保正确的执行顺序。</p>
</li>
</ul>
<hr>
<h3 id="defer-以外的其它-script-标签属性"><a href="#defer-以外的其它-script-标签属性"><code>defer</code> 以外的其它 Script 标签属性</a></h3>
<p>除了 <code>defer</code>，<code>&#x3C;script></code> 标签还有一系列非常重要的属性。其中，与 <code>defer</code> 最常比较的是 <code>async</code>。</p>
<h4 id="async异步无序的执行"><a href="#async异步无序的执行"><code>async</code>：异步无序的执行</a></h4>
<p><code>async</code> (asynchronous) 属性也用于实现非阻塞的脚本加载，但它和 <code>defer</code> 有一个关键区别。</p>
<ol>
<li>
<p><strong>并行下载</strong>：和 <code>defer</code> 一样，<code>async</code> 脚本的下载过程与 HTML 解析并行，不阻塞页面渲染。</p>
</li>
<li>
<p><strong>立即执行</strong>：<strong>关键区别来了</strong>。一旦 <code>async</code> 脚本下载完成，它会<strong>立即执行</strong>，而不管 HTML 文档是否解析完毕。这可能会暂停 HTML 的解析。</p>
</li>
<li>
<p><strong>无序执行</strong>：如果有多个 <code>async</code> 脚本，它们的执行顺序是不可预测的。哪个先下载完，哪个就先执行。这被称为“竞争”或“竞速”状态。</p>
</li>
</ol>
<p>async 的适用场景：</p>
<p>它适用于那些完全独立、不依赖于其他脚本、也不需要操作 DOM 的脚本。最典型的例子：</p>
<ul>
<li>
<p><strong>网站分析脚本</strong> (如 Google Analytics)</p>
</li>
<li>
<p><strong>广告脚本</strong></p>
</li>
<li>
<p><strong>各种追踪器</strong></p>
</li>
</ul>
<p>这些脚本的共同点是：它们自己运行自己的，不关心页面结构，也不影响其他功能。早点执行或晚点执行都无所谓。</p>
<hr>
<h3 id="对比与抉择defer-vs-async-vs-默认"><a href="#对比与抉择defer-vs-async-vs-默认">对比与抉择：<code>defer</code> vs <code>async</code> vs 默认</a></h3>
<p>为了彻底清晰，我们用一个表格来总结：</p>
<table>
<thead>
<tr>
<th>属性</th>
<th>HTML解析时下载</th>
<th>阻塞HTML解析？</th>
<th>脚本执行时机</th>
<th>执行顺序</th>
<th>DOM是否完备</th>
<th>最佳用途</th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>无</strong></td>
<td>否（暂停解析去下载）</td>
<td><strong>是</strong>（下载和执行都阻塞）</td>

<td>严格按顺序</td>
<td>不一定，取决于脚本位置</td>
<td><strong>基本不推荐使用</strong></td>
</tr>
<tr>
<td><code>defer</code></td>
<td>是（并行下载）</td>
<td><strong>否</strong></td>
<td>HTML解析完毕后，<code>DOMContentLoaded</code>前</td>
<td><strong>严格按顺序</strong></td>

</tr>

<h4 id="我的选择建议"><a href="#我的选择建议">我的选择建议</a></h4>
<p>对于现代 Web 开发，你应该遵循以下原则：</p>
<ol>
<li>
<p><strong><code>defer</code> 是首选和默认选项</strong>：对于所有对页面功能至关重要的脚本（比如你的应用主逻辑 <code>app.js</code>、UI框架 <code>vue.js</code> 等），都应该使用 <code>defer</code> 并将它们放在 <code>&#x3C;head></code> 中。这既能获得最佳的加载性能，又能保证脚本逻辑的稳定可靠。</p>
</li>
<li>
<p><strong><code>async</code> 用于特定场景</strong>：只在你加载那些完全独立的第三方脚本时，才考虑使用 <code>async</code>。</p>
</li>
<li>
<p><strong>避免无属性的脚本</strong>：尽量避免在 <code>&#x3C;head></code> 中使用无属性的 <code>&#x3C;script></code>。如果你确实有极少数情况，需要在页面渲染的某个特定点立即执行一段代码（比如一个修复早期渲染问题的 polyfill），可以考虑将其放在 <code>&#x3C;body></code> 的末尾，但这通常不如 <code>defer</code> 优雅。</p>
</li>
</ol>
<hr>
<h3 id="其他重要属性"><a href="#其他重要属性">其他重要属性</a></h3>
<p>除了 <code>defer</code> 和 <code>async</code>，还有一些属性你也应该了解。</p>
<ul>
<li>
<p><strong><code>src</code> (source)</strong></p>
<ul>
<li>
<p><strong>作用</strong>：指定外部脚本文件的 URL。这是最基本、最常用的属性。如果一个 <code>&#x3C;script></code> 标签没有 <code>src</code> 属性，那么浏览器会将其视为内联脚本，并执行标签内的代码。</p>
</li>
<li>
<p><strong>示例</strong>：<code>&#x3C;script src="/path/to/script.js">&#x3C;/script></code></p>
</li>
</ul>
</li>
<li>
<p><strong><code>type</code></strong></p>
<ul>
<li>
<p><strong>作用</strong>：声明脚本的 MIME 类型。在 HTML5 中，其默认值是 <code>text/javascript</code>，所以你通常可以省略它。</p>
</li>
<li>
<p><strong>重要特例 <code>type="module"</code></strong>：这是现代 JavaScript 的一个重要属性。它告诉浏览器这个脚本是一个 ES 模块。</p>
<ul>
<li>
<p>模块脚本默认行为类似 <code>defer</code>（异步加载，DOM 解析后执行），无需再额外添加 <code>defer</code>。</p>
</li>
<li>
<p>它有自己的顶级作用域，不会污染全局命名空间。</p>
</li>
<li>
<p>可以使用 <code>import</code> 和 <code>export</code> 关键字。</p>
</li>
</ul>
</li>
<li>
<p><strong>示例</strong>：<code>&#x3C;script type="module" src="main.mjs">&#x3C;/script></code></p>
</li>
</ul>
</li>
<li>
<p><strong><code>nomodule</code></strong></p>
<ul>
<li>
<p><strong>作用</strong>：这是一个布尔属性，用于为不支持 ES 模块的老旧浏览器提供一个回退脚本。支持 <code>type="module"</code> 的现代浏览器会忽略带有 <code>nomodule</code> 属性的脚本。</p>
</li>
<li>
<p><strong>示例</strong>：</p>
<p>HTML</p>

<p><strong><code>integrity</code> (Subresource Integrity - SRI)</strong></p>
<ul>
<li>
<p><strong>作用</strong>：一个重要的安全特性。它允许你提供一个文件内容的哈希值。浏览器在下载脚本后，会计算其哈希值并与你提供的值进行比对。如果不匹配，浏览器将拒绝执行该脚本。</p>
</li>
<li>
<p><strong>为什么需要？</strong>：这可以防止 CDN（内容分发网络）被攻击者篡改后，向你的用户提供恶意脚本。</p>

</li>
<li>
<p><code>anonymous</code>：默认值。发送请求时不携带任何身份凭证（如 Cookie 或 HTTP 认证）。</p>
</li>
<li>
<p><code>use-credentials</code>：发送请求时会携带身份凭证。</p>
</li>
<li>
<p><strong>为什么需要？</strong>：如果你想捕获跨域脚本中发生的错误详情（而不是只得到一个模糊的 "Script error."），你就必须设置 <code>crossorigin="anonymous"</code>。</p>
</li>
</ul>

<li>
<p><strong><code>referrerpolicy</code></strong></p>

<li><strong>作用</strong>：一个更精细的隐私控制属性。它用于控制在请求脚本时，HTTP <code>Referer</code> 头部会发送哪些信息。例如，<code>no-referrer</code> 表示完全不发送来源信息。</li>
