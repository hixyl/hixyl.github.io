19:T8859,<p>实现一个 JavaScript 的异步模块定义（AMD）加载器是一个非常好的编程练习，它能让你深入理解模块化、依赖管理和异步编程的核心。</p>
<p>我们将从第一性原理出发，逐步构建一个简单但功能完备的 AMD 加载器。整个过程会分为几个核心步骤：</p>
<ol>
<li>
<p><strong>定义核心数据结构</strong>：我们需要一个地方来存储和管理所有模块的状态。</p>
</li>
<li>
<p><strong>实现 <code>define</code> 函数</strong>：模块的“定义”者，用于声明一个模块及其依赖。</p>
</li>
<li>
<p><strong>实现 <code>require</code> 函数</strong>：模块的“使用者”，用于启动模块的加载和执行。</p>
</li>
<li>
<p><strong>实现依赖加载与执行</strong>：这是加载器的核心，负责下载脚本、解决依赖关系并执行模块代码。</p>
</li>
</ol>
<h3 id="核心思想"><a href="#核心思想">核心思想</a></h3>
<p>AMD 的核心是解决两个问题：</p>
<ol>
<li>
<p><strong>异步加载</strong>：在浏览器中，加载 JavaScript 文件是异步的，我们不能阻塞页面渲染。</p>
</li>
<li>
<p><strong>依赖管理</strong>：一个模块可能依赖其他多个模块，必须确保所有依赖都加载完成后，才执行当前模块。</p>
</li>
</ol>
<p>解决方案是通过一个全局的 <code>define</code> 函数，让每个模块文件都能“注册”自己，包括它的依赖项和工厂函数。加载器则负责解析这些注册信息，构建依赖图，并按正确的顺序加载和执行它们。</p>
<hr>
<h3 id="第一步基础架构和数据管理"><a href="#第一步基础架构和数据管理">第一步：基础架构和数据管理</a></h3>
<p>我们需要一个闭包来封装加载器的所有内部状态和方法，避免污染全局作用域。同时，需要一个对象来缓存所有模块的信息。</p>
<p>一个模块通常有以下几种状态：</p>
<ul>
<li>
<p><strong>LOADING</strong>：正在从服务器加载脚本文件。</p>
</li>
<li>
<p><strong>LOADED</strong>：脚本已加载，但依赖项可能还未就绪。</p>
</li>
<li>
<p><strong>DEFINED</strong>：模块的工厂函数已执行，模块已完全可用。</p>
</li>
</ul>
<p>JavaScript</p>

<h3 id="第二步实现-define-函数"><a href="#第二步实现-define-函数">第二步：实现 <code>define</code> 函数</a></h3>
<p><code>define</code> 函数是给模块文件使用的。当一个 JS 文件被加载执行时，它应该调用全局的 <code>define</code> 函数来注册自己。</p>
<p><code>define(id?, dependencies?, factory)</code> 有三种常见的调用形式：</p>
<ul>
<li>
<p><code>define(factory)</code>: 匿名模块，无依赖。</p>
</li>
<li>
<p><code>define(dependencies, factory)</code>: 匿名模块，有依赖。</p>
</li>
<li>
<p><code>define(id, dependencies, factory)</code>: 命名模块，有依赖。（我们的简化版加载器主要处理匿名模块，这是最常见的情况）。</p>
</li>
</ul>
<p>JavaScript</p>
<pre><code class="hljs language-ini">// (继续在闭包内)

    checkDependencies(mod)<span class="hljs-comment">;</span>

<h3 id="第三步实现-require-函数和脚本加载"><a href="#第三步实现-require-函数和脚本加载">第三步：实现 <code>require</code> 函数和脚本加载</a></h3>
<p><code>require</code> 是用户启动加载过程的入口。它和 <code>define</code> 非常相似，但不定义模块，而是直接在依赖加载后执行一个回调。</p>
<p>JavaScript</p>
<pre><code class="hljs language-ini">// (继续在闭包内)

function require(dependencies, callback) {
    // require可以看作是一个临时的、匿名的模块
    const <span class="hljs-attr">tempModule</span> = new Module(<span class="hljs-string">'require_module_'</span> + Date.now())<span class="hljs-comment">;</span>
    <span class="hljs-attr">tempModule.dependencies</span> = dependencies<span class="hljs-comment">;</span>
    <span class="hljs-attr">tempModule.factory</span> = function() {
        // 当所有依赖都准备好后，执行回调
        callback.apply(null, arguments)<span class="hljs-comment">;</span>
    }<span class="hljs-comment">;</span>
    <span class="hljs-attr">tempModule.status</span> = Status.LOADED<span class="hljs-comment">; // 假装它已加载</span>

    // 立即开始处理它的依赖
    loadDependencies(tempModule)<span class="hljs-comment">;</span>
}

function loadDependencies(mod) {
    mod.dependencies.forEach(<span class="hljs-attr">depId</span> => {
        const <span class="hljs-attr">depModule</span> = getModule(depId)<span class="hljs-comment">;</span>
        if (<span class="hljs-attr">depModule.status</span> === Status.DEFINED) {
            return<span class="hljs-comment">; // 依赖已就绪，跳过</span>
        }

        // 如果依赖正在加载或已加载但未定义，
        // 添加一个回调，当依赖准备好后，回来检查当前模块
        depModule.callbacks.push(() => {
            checkDependencies(mod)<span class="hljs-comment">;</span>
        })<span class="hljs-comment">;</span>

        // 如果依赖是第一次遇到，则加载它
        if (<span class="hljs-attr">depModule.status</span> === Status.LOADING) {
             // 避免重复加载脚本
            if (!depModule.isLoadingScript) {
                 <span class="hljs-attr">depModule.isLoadingScript</span> = <span class="hljs-literal">true</span><span class="hljs-comment">;</span>
                 loadScript(depId)<span class="hljs-comment">;</span>
            }
        }
    })<span class="hljs-comment">;</span>

    // 在分发完加载任务后，立即检查一次
    checkDependencies(mod)<span class="hljs-comment">;</span>

function loadScript(id) {

    <span class="hljs-attr">script.onload</span> = () => {
        // 当脚本加载完成时，全局的 define 函数应该刚刚被执行

        onScriptLoad(id)<span class="hljs-comment">;</span>

    <span class="hljs-attr">script.onerror</span> = () => {

    // 这样 define 函数才知道自己属于哪个模块
    <span class="hljs-attr">currentLoadingModule</span> = getModule(id)<span class="hljs-comment">;</span>
    
    document.head.appendChild(script)<span class="hljs-comment">;</span>

    const <span class="hljs-attr">mod</span> = getModule(id)<span class="hljs-comment">;</span>

    loadDependencies(mod)<span class="hljs-comment">;</span>

<h3 id="局限性与改进方向"><a href="#局限性与改进方向">局限性与改进方向</a></h3>
<p>这个实现为了清晰地展示核心原理，省略了很多生产级加载器必备的功能：</p>
<ol>
<li>
<p><strong>错误处理</strong>：脚本加载超时、404错误、模块定义错误等都需要完善的处理机制。</p>
</li>
<li>
<p><strong>路径解析</strong>：目前我们用 <code>id + '.js'</code> 的简单方式。真正的加载器需要能处理相对路径 (<code>./util</code>)、根路径 (<code>/lib</code>) 和复杂的路径映射配置。</p>
</li>
<li>
<p><strong>循环依赖</strong>：如果 A 依赖 B，B又依赖 A，我们的简单实现可能会导致死锁或栈溢出。生产级加载器需要能检测并优雅地处理循环依赖。</p>
</li>
<li>
<p><strong>插件系统</strong>：像 RequireJS 支持 <code>text!</code> 或 <code>json!</code> 这样的插件来加载非 JS 资源。</p>
</li>
<li>
<p><strong>性能优化</strong>：对于打包后的文件（多个 <code>define</code> 在一个文件里），加载逻辑需要调整。</p>
</li>
</ol>
<h3 id="结论与现代前端的视角"><a href="#结论与现代前端的视角">结论与现代前端的视角</a></h3>
<p>亲手实现一个 AMD 加载器是理解前端模块化历史和异步依赖管理绝佳的方式。它所蕴含的思想，如依赖图、回调、状态管理，在现代前端工具中依然以各种形式存在。</p>
<p>然而，对于<strong>当前和未来的新项目</strong>，我的建议是<strong>直接使用原生的 ES Modules (ESM)</strong>。</p>
<p>ESM (<code>import</code>/<code>export</code> 语法) 是现代 JavaScript 的标准，并已在所有主流浏览器和 Node.js 中得到原生支持。它有几个显著优势：</p>
<ul>
<li>
<p><strong>静态分析</strong>：<code>import</code> 和 <code>export</code> 必须在顶层，这使得打包工具（如 Webpack, Rollup, Vite）可以在编译时进行高效的摇树优化（Tree Shaking），移除未使用的代码。</p>
</li>
<li>
<p><strong>语法简洁</strong>：相比 <code>define([...], function(...) {})</code> 的回调地狱风格，<code>import</code> 语法更直观、更易读写。</p>
</li>
<li>
<p><strong>标准</strong>：作为语言标准，它拥有最好的生态支持和未来保障。</p>
</li>
</ul>
<p>总而言之，将实现 AMD 加载器视为一次宝贵的学习经历，但在实际开发中，拥抱 ES Modules 是更明智、更高效的选择。</p>
<p>你好！你的问题非常棒，正中这个 AMD 加载器实现的核心与精妙之处。<code>define</code> 函数和 <code>currentLoadingModule</code> 的交互确实是理解整个异步流程的关键。</p>
<p>让我从第一性原理出发，一步步为你拆解这个流程，解答你的困惑。</p>
<hr>
<h3 id="核心解惑define-并不主动工作而是被动响应"><a href="#核心解惑define-并不主动工作而是被动响应">核心解惑：<code>define</code> 并不“主动”工作，而是“被动”响应</a></h3>
<p>首先要建立一个核心观念：<code>define</code> 函数本身不负责创建模块对象，也不负责加载任何文件。</p>
<ul>
<li><strong><code>define</code> 的角色</strong>：它像一个“登记员”。当一个模块JS文件被浏览器加载并执行时，这个文件里的 <code>define(...)</code> 代码就会运行。<code>define</code> 函数的作用就是接收这个模块的信息（依赖、工厂函数），然后把它登记到<strong>一个已经存在、并且正在等待被登记的模块对象</strong>上。</li>
</ul>
<p>那么，这个“已经存在、正在等待登记的模块对象”是谁创建的？<code>currentLoadingModule</code>又是在哪里被赋值的呢？</p>
<p>答案是：这一切都始于 <code>require</code>，并通过 <code>loadScript</code> 函数串联起来。</p>
<h3 id="完整流程追踪从-require-到-define"><a href="#完整流程追踪从-require-到-define">完整流程追踪：从 <code>require</code> 到 <code>define</code></a></h3>
<p>我们以 <code>require(['modules/main'], ...)</code> 这个入口为例，看看代码的执行顺序：</p>
<p><strong>第 1 步：<code>require</code> 启动</strong></p>
<ol>
<li>
<p>你调用 <code>require(['modules/main'], callback)</code>.</p>
</li>
<li>
<p><code>require</code> 函数创建一个临时的、匿名的模块来管理这次调用。</p>
</li>
<li>
<p>最关键的是，它调用 <code>loadDependencies(['modules/main'])</code>，开始加载依赖。</p>
</li>
</ol>
<p><strong>第 2 步：<code>loadDependencies</code> 和 <code>getModule</code> 首次登场</strong></p>
<ol>
<li>
<p><code>loadDependencies</code> 遍历依赖数组，拿到第一个依赖ID：<code>'modules/main'</code>。</p>
</li>
<li>
<p>它调用 <code>getModule('modules/main')</code>。</p>
</li>
<li>
<p>因为这是第一次遇到 <code>'modules/main'</code>，缓存 <code>modules</code> 中没有它，所以 <code>getModule</code> 会执行 <code>new Module('modules/main')</code>。</p>
<ul>
<li><strong>回答你的问题</strong>：<strong><code>new Module</code> 在这里发生！</strong> 一个代表 <code>'modules/main'</code> 的模块对象被创建了，它的初始状态是 <code>LOADING</code>。</li>
</ul>
</li>
<li>
<p><code>loadDependencies</code> 发现这个新模块的状态是 <code>LOADING</code>，并且还没有开始加载脚本 (<code>isLoadingScript</code> 是 <code>false</code>)。于是，它决定去加载脚本。</p>
</li>
</ol>
<p><strong>第 3 步：<code>loadScript</code> —— 赋值 <code>currentLoadingModule</code> 的关键时刻</strong></p>
<ol>
<li>
<p><code>loadDependencies</code> 调用 <code>loadScript('modules/main')</code>。</p>
</li>
<li>
<p><code>loadScript</code> 创建一个 <code>&#x3C;script></code> 标签，<code>src</code> 指向 <code>modules/main.js</code>。</p>
</li>
<li>
<p><strong>在把这个 script 标签插入到 HTML 之前，执行了最核心的一行代码：</strong></p>
<p>JavaScript</p>
<pre><code class="hljs language-bash">currentLoadingModule = getModule(<span class="hljs-built_in">id</span>); // <span class="hljs-built_in">id</span> 是 <span class="hljs-string">'modules/main'</span>
</code></pre>
<ul>
<li><strong>回答你的问题</strong>：<strong><code>currentLoadingModule</code> 在这里被赋值了！</strong> 它现在指向了我们在第 2 步创建的那个 <code>'modules/main'</code> 模块对象。此时，这个全局（闭包内）变量就像一个标记，告诉整个加载器：“我接下来要处理的 <code>define</code>，就是为 <code>'modules/main'</code> 服务的。”</li>
</ul>
</li>
<li>
<p><code>loadScript</code> 随后将 <code>&#x3C;script></code> 标签添加到 <code>&#x3C;head></code> 中。浏览器开始异步下载 <code>modules/main.js</code>。</p>
</li>
</ol>
<p><strong>第 4 步：<code>define</code> 被动执行</strong></p>
<ol>
<li>
<p>浏览器下载并执行 <code>modules/main.js</code> 文件。</p>
</li>
<li>
<p>该文件中的代码 <code>define(['modules/math'], function(math) { ... });</code> 被执行。</p>
</li>
<li>
<p>此时，我们的全局 <code>define</code> 函数被调用了。</p>
</li>
<li>
<p>在 <code>define</code> 函数内部，它读取 <code>currentLoadingModule</code>。因为第 3 步的赋值，<code>currentLoadingModule</code> <strong>不再是 null</strong>，它就是 <code>'modules/main'</code> 的模块对象。</p>
</li>
<li>
<p><code>define</code> 函数将依赖 <code>['modules/math']</code> 和工厂函数 <code>function(math) {...}</code> 存入 <code>currentLoadingModule</code> 这个对象中，并将其状态更新为 <code>LOADED</code>（表示脚本加载完成，信息已登记，但依赖尚未解决）。</p>
</li>
</ol>
<p><strong>第 5 步：<code>onload</code> 与递归加载</strong></p>
<ol>
<li>
<p><code>modules/main.js</code> 文件执行完毕后，<code>script</code> 标签的 <code>onload</code> 事件触发，调用 <code>onScriptLoad('modules/main')</code>。</p>
</li>
<li>
<p><code>onScriptLoad</code> 拿到 <code>'modules/main'</code> 模块对象，发现它的依赖 <code>['modules/math']</code> 还没处理，于是再次调用 <code>loadDependencies(['modules/math'])</code>。</p>
</li>
<li>
<p>流程回到第 2 步，但这次是为 <code>'modules/math'</code> 创建模块对象、在 <code>loadScript</code> 中设置 <code>currentLoadingModule</code>、加载 <code>math.js</code>、执行 <code>math.js</code> 中的 <code>define</code>... 如此循环往复，直到所有依赖链都加载并定义完毕。</p>
</li>
</ol>
<h3 id="总结与图解"><a href="#总结与图解">总结与图解</a></h3>
<p>简单来说，这是一个“安排与响应”的机制：</p>
<ol>
<li>
<p><strong>安排 (<code>require</code> -> <code>loadDependencies</code> -> <code>loadScript</code>)</strong>:</p>
<ul>
<li>
<p>“我需要一个叫 <code>main</code> 的模块。”</p>
</li>
<li>
<p>创建 <code>main</code> 的模块对象（状态：LOADING）。</p>
</li>
<li>
<p><strong>在加载 <code>main.js</code> 脚本前，用 <code>currentLoadingModule</code> 全局“记住” <code>main</code> 这个模块对象。</strong></p>
</li>
<li>
<p>命令浏览器去加载 <code>main.js</code>。</p>
</li>
</ul>
</li>
<li>
<p><strong>响应 (<code>define</code>)</strong>:</p>
<ul>
<li>
<p><code>main.js</code> 加载并执行。</p>
</li>
<li>
<p>它里面的 <code>define</code> 函数被调用。</p>
</li>
<li>
<p><code>define</code> 说：“我知道了，我这些信息（依赖和工厂函数）是属于 <code>currentLoadingModule</code>（也就是 <code>main</code> 模块）的。”</p>
</li>
<li>
