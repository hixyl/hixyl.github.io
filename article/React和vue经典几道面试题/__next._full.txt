<h2 id="react-核心面试题与深度解析"><a href="#react-核心面试题与深度解析">React 核心面试题与深度解析</a></h2>
<h3 id="1-react-的渲染机制是怎样的fiber-架构解决了什么问题"><a href="#1-react-的渲染机制是怎样的fiber-架构解决了什么问题">1. React 的渲染机制是怎样的？Fiber 架构解决了什么问题？</a></h3>
<p><strong>解析：</strong></p>
<p>这道题考察对 React 底层运作的宏观认知。</p>
<p>在 React 16 之前，React 的渲染过程是同步的。当组件树极其庞大时，虚拟 DOM 的 Diff 运算会长时间占用主线程，导致浏览器无法及时响应用户的输入或动画渲染，产生卡顿。</p>
<p>Fiber 架构本质上实现了<strong>时间分片（Time Slicing）和可中断渲染</strong>。</p>
<ul>
<li>
<p>将耗时的渲染任务拆分成多个小的工作单元（Fiber 节点）。</p>
</li>
<li>
<p>利用浏览器的空闲时间（<code>requestIdleCallback</code> 的 polyfill 实现）来执行这些单元。</p>
</li>
<li>
<p>如果执行过程中有更高优先级的任务（如用户输入），React 会暂停当前渲染，优先处理高优先级任务，之后再恢复或重新计算。</p>
</li>
</ul>

<h3 id="2-请解释-react-hooks-中的闭包陷阱stale-closures并说明如何解决"><a href="#2-请解释-react-hooks-中的闭包陷阱stale-closures并说明如何解决">2. 请解释 React Hooks 中的“闭包陷阱”（Stale Closures），并说明如何解决？</a></h3>
<p><strong>解析：</strong></p>
<p>这是实际业务中最容易踩坑的场景。Hooks 的运行高度依赖 JavaScript 的闭包机制。当我们在 <code>useEffect</code> 或定时器中引用了某个 state 时，如果依赖数组设置不当，闭包中捕获的将是“历史版本”的 state，而非最新值。</p>
<p><strong>错误示例（闭包陷阱）：</strong></p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">import</span> { useState, useEffect } <span class="hljs-keyword">from</span> <span class="hljs-string">'react'</span>;

<span class="hljs-keyword">function</span> <span class="hljs-title function_">Counter</span>(<span class="hljs-params"></span>) {
  <span class="hljs-keyword">const</span> [count, setCount] = <span class="hljs-title function_">useState</span>(<span class="hljs-number">0</span>);

  <span class="hljs-title function_">useEffect</span>(<span class="hljs-function">() =></span> {
    <span class="hljs-keyword">const</span> timer = <span class="hljs-built_in">setInterval</span>(<span class="hljs-function">() =></span> {
      <span class="hljs-comment">// 这里的 count 永远是初次渲染时的 0</span>
      <span class="hljs-title function_">setCount</span>(count + <span class="hljs-number">1</span>); 
    }, <span class="hljs-number">1000</span>);
    <span class="hljs-keyword">return</span> <span class="hljs-function">() =></span> <span class="hljs-built_in">clearInterval</span>(timer);

}
</code></pre>
<p><strong>解决方案：</strong></p>
<p>解决此问题的核心在于打破陈旧的作用域，或者让 React 知道你需要最新的状态。</p>
<ul>
<li>
<p><strong>方案一：使用函数式更新（推荐）。</strong> <code>setCount</code> 接收一个函数，该函数的参数总是最新的 state。</p>
</li>
<li>
<p><strong>方案二：利用 <code>useRef</code>。</strong> <code>useRef</code> 的引用在整个组件生命周期内保持不变，可以用来存储最新值。</p>
</li>
</ul>
<p>JavaScript</p>

<span class="hljs-keyword">const</span> countRef = <span class="hljs-title function_">useRef</span>(count);
<span class="hljs-title function_">useEffect</span>(<span class="hljs-function">() =></span> {
  countRef.<span class="hljs-property">current</span> = count;
}, [count]);

<h3 id="3-react-的性能优化手段有哪些何时应该避免使用-usememousecallback"><a href="#3-react-的性能优化手段有哪些何时应该避免使用-usememousecallback">3. React 的性能优化手段有哪些？何时应该避免使用 useMemo/useCallback？</a></h3>
<p><strong>解析：</strong></p>
<p>很多开发者习惯性地为所有函数和计算结果包裹 <code>useMemo</code> 和 <code>useCallback</code>，这反而会导致性能下降。</p>
<ul>
<li>
<p><code>React.memo</code>：用于拦截父组件渲染引发的子组件不必要的渲染。进行的是浅比较（Shallow Compare）。</p>
</li>
<li>
<p><code>useMemo</code>：缓存昂贵的计算结果。</p>
</li>
<li>
<p><code>useCallback</code>：缓存函数引用，通常配合 <code>React.memo</code> 使用，防止子组件因每次接收新的函数引用而重新渲染。</p>
</li>
</ul>
<p><strong>反直觉考量（边界条件）：</strong></p>
<p>缓存机制本身是有成本的（依赖数组的对比、闭包的创建、内存占用）。如果计算过程非常简单（例如简单的加减法或少量的数组映射），使用 <code>useMemo</code> 的开销往往大于重新计算的开销。只有当计算复杂度极高，或该变量作为其他深层级 Hook 的依赖时，缓存才具有正向收益。</p>

<h2 id="vue-核心面试题与深度解析"><a href="#vue-核心面试题与深度解析">Vue 核心面试题与深度解析</a></h2>
<h3 id="1-vue-2-和-vue-3-的响应式原理有何本质区别为什么要做出这种改变"><a href="#1-vue-2-和-vue-3-的响应式原理有何本质区别为什么要做出这种改变">1. Vue 2 和 Vue 3 的响应式原理有何本质区别？为什么要做出这种改变？</a></h3>
<p><strong>解析：</strong></p>
<p>这是 Vue 最核心的基础题，涉及底层 API 的变更及随之带来的架构红利。</p>
<ul>
<li>
<p><strong>Vue 2（Object.defineProperty）：</strong></p>
<p>通过递归遍历对象的所有属性，将其转化为 getter/setter 来实现拦截。</p>
<p><em>缺陷</em>：必须在初始化时知道属性的存在，无法原生监听到对象属性的添加和删除（需要 <code>$set</code>、<code>$delete</code>）；无法完美监听数组的索引修改和长度变化（需要重写数组原型方法）；深层对象初始化时递归开销大。</p>
</li>
<li>
<p><strong>Vue 3（Proxy）：</strong></p>
<p>使用 ES6 的 <code>Proxy</code> 直接代理整个对象，而非修改对象属性。</p>
<p><em>优势</em>：原生支持监听属性的新增和删除；原生支持监听数组变化；支持 Map、Set 等数据结构；惰性响应（只有在访问到深层属性时，才会将其转化为响应式，极大提升了初始渲染性能）。</p>
</li>
</ul>
<p><strong>代码演示（简易版 Proxy 响应式）：</strong></p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">function</span> <span class="hljs-title function_">reactive</span>(<span class="hljs-params">target</span>) {
  <span class="hljs-keyword">return</span> <span class="hljs-keyword">new</span> <span class="hljs-title class_">Proxy</span>(target, {
    <span class="hljs-title function_">get</span>(<span class="hljs-params">target, key, receiver</span>) {
      <span class="hljs-comment">// 收集依赖 (Track)</span>
      <span class="hljs-variable language_">console</span>.<span class="hljs-title function_">log</span>(<span class="hljs-string">`读取了属性：<span class="hljs-subst">${key}</span>`</span>);
      <span class="hljs-keyword">return</span> <span class="hljs-title class_">Reflect</span>.<span class="hljs-title function_">get</span>(target, key, receiver);
    },

<h3 id="2-为什么-vue-3-要引入-composition-api组合式-api它比-options-api-好在哪里"><a href="#2-为什么-vue-3-要引入-composition-api组合式-api它比-options-api-好在哪里">2. 为什么 Vue 3 要引入 Composition API（组合式 API）？它比 Options API 好在哪里？</a></h3>
<p><strong>解析：</strong></p>
<p>单纯回答“为了代码组织更好”是不够的，需要从代码复用和逻辑解耦的层面深入。</p>
<p>在 Vue 2 的 Options API（<code>data</code>, <code>methods</code>, <code>computed</code>, <code>watch</code>）中，当一个组件极其复杂时，同一个业务逻辑的代码会被打散在不同的选项中（碎片化）。这就导致阅读和重构代码时，必须在文件上下反复跳跃。</p>
<p>Composition API 允许我们基于<strong>逻辑关注点</strong>来组织代码。更重要的是，它提供了一种极其优雅的<strong>逻辑提取与复用</strong>方式（Custom Hooks / Composables），彻底解决了 Vue 2 中 <code>mixin</code> 带来的命名冲突和数据来源不清晰的问题。</p>

<h3 id="3-vue-3-在编译层面做了哪些优化compiler-optimizations"><a href="#3-vue-3-在编译层面做了哪些优化compiler-optimizations">3. Vue 3 在编译层面做了哪些优化（Compiler Optimizations）？</a></h3>
<p><strong>解析：</strong></p>
<p>Vue 3 的性能飞跃不仅来自于响应式系统的重写，更来自于编译器在构建时所做的静态分析。Vue 突破了传统 Virtual DOM 的全量对比瓶颈。</p>
<ul>
<li>
<p><strong>静态提升（Static Hoisting）：</strong> Vue 会将不包含任何动态绑定的静态节点在编译阶段提升到渲染函数外部，每次渲染直接复用，避免重新创建对象。</p>
</li>
<li>
<p><strong>补丁标记（Patch Flags）：</strong> 在动态节点上标记出哪些属性是动态的（如只绑定了 class，或只绑定了 text）。在更新时，Diff 算法只需针对带有特定 Flag 的属性进行比对，极大地减少了无用功。</p>
</li>
<li>
<p><strong>Block Tree：</strong> 将模板结构打平，所有的动态节点被收集到一个扁平的数组中。更新时只遍历这个数组，使得 DOM 更新的复杂度与动态节点的数量成正比，而非与整个模板的节点总量成正比。</p>
</li>
</ul>
