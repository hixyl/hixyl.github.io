4:T318e,<p>在 Vue 3 中，自定义 Hooks（官方称为 <strong>Composables</strong>，组合式函数）是利用组合式 API（Composition API）来封装和复用<strong>有状态逻辑</strong>的最佳实践。</p>
<p>它解决了以前 <code>Mixins</code> 存在的命名冲突、来源不清晰以及逻辑分散等痛点。我们可以把一个复杂的组件逻辑拆分成一个个独立的、可测试的微小单元。</p>
<hr>
<h2 id="1-基本结构"><a href="#1-基本结构">1. 基本结构</a></h2>
<p>一个标准的 Composable 通常遵循以下约定：</p>
<ul>
<li>
<p><strong>命名：</strong> 以 <code>use</code> 开头（如 <code>useMouse</code>, <code>useFetch</code>）。</p>
</li>
<li>
<p><strong>输入：</strong> 可以接收参数（可以是普通值、Ref 或 Getter）。</p>
</li>
<li>
<p><strong>输出：</strong> 通常返回一个包含 <code>ref</code> 或 <code>function</code> 的对象（方便解构，但也需注意解构后会丢失响应性的坑，通常建议返回 Ref 对象）。</p>
</li>
</ul>
<hr>
<h2 id="2-核心示例追踪鼠标位置"><a href="#2-核心示例追踪鼠标位置">2. 核心示例：追踪鼠标位置</a></h2>
<p>这是最经典的一个例子，展示了如何封装状态和生命周期钩子。</p>

<h2 id="3-进阶技巧与最佳实践"><a href="#3-进阶技巧与最佳实践">3. 进阶技巧与最佳实践</a></h2>
<p>为了写出高质量的 Composables，建议关注以下几个维度：</p>
<h3 id="a-灵活的输入参数"><a href="#a-灵活的输入参数">A. 灵活的输入参数</a></h3>
<p>优秀的 Hooks 应该既支持传入普通值，也支持传入 Ref。可以使用 <code>toValue</code>（Vue 3.3+）来统一处理。</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">import</span> { toValue, watchEffect } <span class="hljs-keyword">from</span> <span class="hljs-string">'vue'</span>

<span class="hljs-keyword">export</span> <span class="hljs-keyword">function</span> <span class="hljs-title function_">useData</span>(<span class="hljs-params">url</span>) {
  <span class="hljs-title function_">watchEffect</span>(<span class="hljs-function">() =></span> {
    <span class="hljs-comment">// 无论 url 是字符串还是 ref，toValue 都能拿到具体的值</span>
    <span class="hljs-keyword">const</span> currentUrl = <span class="hljs-title function_">toValue</span>(url)
    <span class="hljs-comment">// 执行 fetch 逻辑...</span>
  })
}
</code></pre>
<h3 id="b-始终返回对象而非数组"><a href="#b-始终返回对象而非数组">B. 始终返回对象而非数组</a></h3>
<p>虽然 React Hooks 习惯返回数组（如 <code>const [val, setVal] = useState()</code>），但在 Vue 中<strong>推荐返回对象</strong>。这样在解构时可以按需取值，且不需要严格遵守顺序。</p>
<h3 id="c-资源清理"><a href="#c-资源清理">C. 资源清理</a></h3>
<p>如果你的 Hook 开启了定时器、网络请求或 DOM 事件监听，务必在 <code>onUnmounted</code> 中清理，防止内存泄漏。</p>
<h3 id="d-嵌套使用"><a href="#d-嵌套使用">D. 嵌套使用</a></h3>
<p>Composables 内部可以调用其他的 Composables。这种“套娃”能力使得你可以通过组合小的逻辑单元来构建极其复杂的业务逻辑。</p>
<hr>
<h2 id="4-为什么选择-composables-而不是-mixins"><a href="#4-为什么选择-composables-而不是-mixins">4. 为什么选择 Composables 而不是 Mixins？</a></h2>
<p>如果你在犹豫是否要重构旧代码，下表对比了它们的本质区别：</p>
<table>
<thead>
<tr>
<th><strong>特性</strong></th>
<th><strong>Mixins</strong></th>
<th><strong>Composables (Hooks)</strong></th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>属性来源</strong></td>
<td>隐式（不知道属性来自哪个 Mixin）</td>
<td><strong>显式</strong>（通过 import 和解构明确来源）</td>
</tr>
<tr>
<td><strong>命名冲突</strong></td>
<td>容易冲突（多个 Mixin 拥有同名变量）</td>
<td><strong>无冲突</strong>（可以在解构时重命名）</td>
</tr>
<tr>
<td><strong>逻辑组织</strong></td>
<td>碎片化（按生命周期拆分）</td>
<td><strong>集中式</strong>（将同一功能的逻辑聚拢在一起）</td>
</tr>
<tr>
<td><strong>类型支持</strong></td>
<td>较差</td>
<td><strong>优秀</strong>（原生支持 TypeScript）</td>
</tr>
</tbody>
</table>
<hr>
<h2 id="我的建议"><a href="#我的建议">我的建议</a></h2>
<p>在实际开发中，<strong>不要为了拆分而拆分</strong>。如果逻辑只在一个组件中使用且代码量适中，直接写在组件内即可。当你发现某个逻辑（如：分页处理、表单校验、防抖节流、第三方库初始化）在多个地方出现时，才是抽象成 Composable 的最佳时机。</p>
<p>此外，在编写涉及异步请求的 Hooks 时，建议利用 Vue 的异步组件和 <code>Suspense</code> 特性，或者像 <code>VueUse</code> 库那样返回 <code>isLoading</code> 和 <code>error</code> 状态。</p>
<hr>
<h1 id="vue-composables-和-react-hooks-的区别"><a href="#vue-composables-和-react-hooks-的区别">Vue Composables 和 React Hooks 的区别</a></h1>
<p>虽然 Vue 3 的 Composables 在命名上致敬了 React Hooks（都以 <code>use</code> 开头），但如果你拆开“引擎盖”看一眼，会发现它们的工作原理其实是两种完全不同的哲学。</p>
<p>简单来说：<strong>React Hooks 是“快照”，而 Vue Composables 是“连接”。</strong></p>
<hr>
<h2 id="1-执行机制一次性-vs-反复运行"><a href="#1-执行机制一次性-vs-反复运行">1. 执行机制：一次性 vs. 反复运行</a></h2>
<p>这是两者最本质的区别，决定了你编写代码时的心智负担。</p>
<ul>
<li>
<p><strong>React Hooks：</strong> 每当组件状态改变，整个函数组件会<strong>重新执行</strong>。这意味着所有的 Hooks 也会跟着重新运行。为了防止性能问题或无限循环，你必须小心翼翼地使用 <code>useMemo</code>、<code>useCallback</code> 和依赖数组。</p>
</li>
<li>
<p><strong>Vue Composables：</strong> <code>setup()</code>（或 <code>&#x3C;script setup></code>）只在组件创建时<strong>执行一次</strong>。它建立了一套响应式的追踪系统。后续的更新是由数据直接触发对应的 DOM 局部变更，而不是重新运行整个逻辑块。</p>
</li>
</ul>
<hr>
<h2 id="2-闭包与心智负担"><a href="#2-闭包与心智负担">2. 闭包与“心智负担”</a></h2>
<p>如果你写过 React，一定被“过时闭包（Stale Closures）”坑过。</p>
<ul>
<li>
<p><strong>React 的困境：</strong> 由于函数反复执行，如果你在某个异步回调里访问状态，拿到的可能是“旧帧”里的数据。</p>
</li>
<li>
<p><strong>Vue 的优势：</strong> 因为代码只运行一次，且数据是基于 <strong>Proxy</strong> 的引用，你拿到的永远是那个响应式对象的最新值。你不需要担心闭包捕获了旧变量，因为“变量”本身就是响应式的容器。</p>
</li>
</ul>
<hr>
<h2 id="3-调用规则的约束"><a href="#3-调用规则的约束">3. 调用规则的约束</a></h2>
<p>React 有著名的“Hooks 规则”，而 Vue 几乎没有。</p>
<table>
<thead>
<tr>
<th><strong>场景</strong></th>
<th><strong>React Hooks</strong></th>
<th><strong>Vue Composables</strong></th>
</tr>
</thead>
<tbody>
<tr>
<td><strong>循环/条件判断</strong></td>
<td><strong>严禁使用</strong>。Hooks 必须严格按照固定的顺序调用。</td>
<td><strong>允许使用</strong>。因为逻辑只在初始化时运行一次，不依赖调用顺序。</td>
</tr>
<tr>
<td><strong>手动依赖管理</strong></td>
<td>必须在 <code>useEffect</code> 等后方手动维护 <code>[dep]</code> 数组。</td>
<td><strong>自动追踪</strong>。<code>watch</code> 和 <code>computed</code> 会自动收集依赖。</td>
</tr>
<tr>
<td><strong>性能优化</strong></td>
<td>需要手动 <code>useCallback</code> 避免子组件无谓重绘。</td>
<td><strong>自动优化</strong>。细粒度的响应式系统确保只有受影响的部分更新。</td>
</tr>
</tbody>
</table>
<hr>
<h2 id="4-数据模型的对比"><a href="#4-数据模型的对比">4. 数据模型的对比</a></h2>
<ul>
<li>
<p><strong>React 是不可变（Immutable）的：</strong> 更新状态必须给出一个全新的对象（<code>setState(prev => ({...prev, count: 1}))</code>）。</p>
</li>
<li>
<p><strong>Vue 是可变（Mutable）的：</strong> 你直接修改 <code>count.value++</code> 即可。Vue 的响应式引擎会自动感知并通知订阅者。</p>
</li>
</ul>

<table>

<h2 id="总结建议"><a href="#总结建议">总结建议</a></h2>
<p>如果你从 React 转到 Vue，最需要改变的习惯就是：<strong>别再纠结“依赖数组”了</strong>。</p>
<p>在 React 中，你是在编写一段“描述当前状态的代码”；而在 Vue 中，你是在编写一个“生产响应式数据的工厂”。</p>
