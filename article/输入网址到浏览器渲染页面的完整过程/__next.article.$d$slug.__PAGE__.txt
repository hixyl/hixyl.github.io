4:T27ba,<p>请完整描述从输入网址到浏览器渲染出页面，发生了什么？</p>
<hr>
<p>这是一道非常经典且能考察前端基本功深度和广度的面试题。为了把这个过程讲透，我们需要从网络协议层到浏览器内核渲染层进行逐一拆解。</p>
<p>以下是从输入 URL 到页面最终呈现的完整链路：</p>
<hr>
<h3 id="第一步url-解析与缓存判断"><a href="#第一步url-解析与缓存判断">第一步：URL 解析与缓存判断</a></h3>
<ol>
<li>
<p><strong>解析 URL</strong>：浏览器会首先分析输入的字符串。提取出协议（如 <code>https</code>）、域名（如 <code>www.example.com</code>）、端口（默认 HTTPS 为 443）、路径和参数。</p>
</li>
<li>
<p><strong>HSTS 检查</strong>：如果网站启用了 HSTS（HTTP 严格传输安全），浏览器会强制将 <code>http</code> 转换为 <code>https</code>。</p>
</li>
<li>
<p><strong>缓存检查</strong>：浏览器会检查是否有本地缓存。</p>
<ul>
<li>
<p>首先检查 <strong>Service Worker</strong>（如果注册过）。</p>
</li>
<li>
<p>接着检查 <strong>Memory Cache</strong>（内存缓存，如页面上加载过的图片）。</p>
</li>
<li>
<p>最后检查 <strong>Disk Cache</strong>（硬盘缓存，检查 HTTP 响应头中的强缓存字段 <code>Expires</code> 和 <code>Cache-Control</code>）。如果强缓存有效，直接返回资源，结束网络请求阶段；如果失效或需协商缓存，则进入下一步。</p>
</li>
</ul>
</li>
</ol>
<h3 id="第二步dns-域名解析"><a href="#第二步dns-域名解析">第二步：DNS 域名解析</a></h3>
<p>由于网络传输需要目标服务器的 IP 地址，而不是域名，因此需要通过 DNS 将域名翻译成 IP 地址。</p>
<p>解析过程是分层查找的：</p>
<ol>
<li>
<p><strong>本地缓存</strong>：浏览器缓存 -> 操作系统缓存（如 <code>hosts</code> 文件） -> 路由器缓存。</p>
</li>
<li>
<p><strong>本地域名服务器 (Local DNS)</strong>：如果本地没有，会向网络运营商 (ISP) 提供的 DNS 服务器发起请求。</p>
</li>
<li>
<p><strong>迭代查询</strong>：如果 Local DNS 也没有，它会向上级发起迭代查询：</p>
<ul>
<li>
<p>根域名服务器（返回顶级域名 <code>.com</code> 的地址）。</p>
</li>
<li>
<p>顶级域名服务器（返回权威域名 <code>example.com</code> 的地址）。</p>
</li>
<li>
<p>权威域名服务器（返回最终的 IP 地址）。</p>
</li>
</ul>
</li>
</ol>
<h3 id="第三步建立-tcp-连接三次握手"><a href="#第三步建立-tcp-连接三次握手">第三步：建立 TCP 连接（三次握手）</a></h3>
<p>拿到 IP 地址后，浏览器会利用底层的 TCP/IP 协议族与服务器建立可靠的连接。TCP 通过“三次握手”来确保双方都有发送和接收的能力。</p>
<ul>
<li>
<p><strong>第一次握手 (SYN)</strong>：客户端生成一个随机序列号 <code>seq = x</code>，并发送一个 SYN 标志位为 1 的数据包给服务器。<strong>（证明客户端有发送能力）</strong></p>
</li>
<li>
<p><strong>第二次握手 (SYN + ACK)</strong>：服务器收到后，回应一个 SYN 为 1、ACK 为 1 的数据包。包含服务器的随机序列号 <code>seq = y</code>，以及确认号 <code>ack = x + 1</code>。<strong>（证明服务器有接收和发送能力）</strong></p>
</li>
<li>
<p><strong>第三次握手 (ACK)</strong>：客户端收到后，发送一个 ACK 为 1 的数据包，<code>ack = y + 1</code>，<code>seq = x + 1</code>。<strong>（证明客户端有接收能力）</strong></p>
<p>至此，双方的收发能力都得到了验证，全双工连接建立。</p>
</li>
</ul>
<h3 id="第四步tlsssl-协商https-加密建立"><a href="#第四步tlsssl-协商https-加密建立">第四步：TLS/SSL 协商（HTTPS 加密建立）</a></h3>
<p>因为 HTTP 是明文传输，极不安全。HTTPS 在 HTTP 和 TCP 之间加入了 TLS/SSL 层。建立 TCP 连接后，还需要进行 TLS 握手来生成对称加密的密钥。</p>
<ol>
<li>
<p><strong>Client Hello</strong>：客户端发送支持的 TLS 版本、加密套件列表和一个<strong>客户端随机数 (Random1)</strong>。</p>
</li>
<li>
<p><strong>Server Hello</strong>：服务器确认 TLS 版本和加密套件，发送数字证书（含服务器公钥）和一个<strong>服务端随机数 (Random2)</strong>。</p>
</li>
<li>
<p><strong>证书验证</strong>：客户端通过操作系统内置的 CA（证书颁发机构）根证书，验证服务器证书的合法性（是否过期、是否被篡改等）。</p>
</li>
<li>
<p><strong>生成预主密钥 (Pre-Master Secret)</strong>：验证通过后，客户端生成第三个随机数**（Pre-Master Secret）**，并用服务器的公钥加密后发送给服务器。</p>
</li>
<li>
<p><strong>生成会话密钥</strong>：服务器用自己的私钥解密出 Pre-Master Secret。此时，客户端和服务器都拥有了三个随机数。双方使用相同的算法，将这三个随机数计算生成最终的**“对称密钥”**（Session Key）。</p>
<p>此后的所有 HTTP 通信，都将使用这个对称密钥进行高效率的加解密。</p>
</li>
</ol>
<h3 id="第五步发送-http-请求与接收响应"><a href="#第五步发送-http-请求与接收响应">第五步：发送 HTTP 请求与接收响应</a></h3>
<ol>
<li>
<p><strong>构建请求</strong>：浏览器根据规范构建 HTTP 请求报文（包含请求行、请求头、请求体），并加上之前提到的协商缓存字段（如 <code>If-Modified-Since</code> 或 <code>If-None-Match</code>）。</p>
</li>
<li>
<p><strong>服务器处理</strong>：请求到达服务器（通常先经过负载均衡、Nginx 反向代理等），后端应用程序处理业务逻辑，可能还会查询数据库。</p>
</li>
<li>
<p><strong>返回响应</strong>：服务器组装 HTTP 响应报文返回。</p>
<ul>
<li>
<p>如果返回状态码 <strong>304 (Not Modified)</strong>，说明协商缓存生效，浏览器直接使用本地缓存。</p>
</li>
<li>
<p>如果返回状态码 <strong>200 (OK)</strong>，说明请求成功，响应体会携带请求的数据（通常是 HTML 文本）。</p>
</li>
</ul>
</li>
</ol>
<h3 id="第六步浏览器解析与渲染"><a href="#第六步浏览器解析与渲染">第六步：浏览器解析与渲染</a></h3>
<p>这是前端开发者最关心的阶段。浏览器内核（如 Chrome 的 Blink 和 V8 引擎）拿到 HTML 文件后，开始将其转化为屏幕上的像素。</p>
<ol>
<li>
<p><strong>构建 DOM 树 (DOM Tree)</strong>：</p>
<ul>
<li>HTML 解析器将字节流转换为字符，再进行词法分析（Tokenize），最终构建出树状的 Document Object Model (DOM)。</li>
</ul>
</li>
<li>
<p><strong>构建 CSSOM 树 (CSSOM Tree)</strong>：</p>
<ul>
<li>
<p>遇到 CSS 资源（<code>&#x3C;link></code> 或 <code>&#x3C;style></code>），会并行发起请求获取（不会阻塞 DOM 解析，但会阻塞渲染）。</p>
</li>
<li>
<p>解析 CSS 生成 CSS Object Model (CSSOM) 树。</p>
</li>
</ul>
</li>
<li>
<p><strong>执行 JavaScript</strong>：</p>
<ul>
<li>遇到 <code>&#x3C;script></code> 标签，HTML 解析会暂停（除非设置了 <code>async</code> 或 <code>defer</code> 属性），等待 JS 下载并执行完毕，因为 JS 有可能修改 DOM 或 CSSOM。</li>
</ul>
</li>
<li>
<p><strong>生成渲染树 (Render Tree)</strong>：</p>
<ul>
<li>
<p>将 DOM 树和 CSSOM 树合并。遍历 DOM 树的可见节点，找到对应的 CSSOM 规则并应用。</p>
</li>
<li>
<p>注意：<code>display: none</code> 的元素不在 Render Tree 中，但 <code>visibility: hidden</code> 的元素会在。</p>
</li>
</ul>
</li>
<li>
<p><strong>布局 / 重排 (Layout / Reflow)</strong>：</p>
<ul>
<li>根据 Render Tree，计算每个节点在屏幕上的确切几何位置和大小。这是个非常消耗性能的过程。</li>
</ul>
</li>
<li>
<p><strong>绘制 / 重绘 (Paint / Repaint)</strong>：</p>
<ul>
<li>遍历 Render Tree，调用操作系统的底层绘图 API，将每个节点转化为屏幕上的实际像素（包括颜色、阴影、文本等）。</li>
</ul>
</li>
<li>
<p><strong>分层与合成 (Compositing)</strong>：</p>
<ul>
<li>
<p>对于复杂的页面，浏览器会将页面分为多个图层（Layer），比如包含 <code>transform: translateZ(0)</code>、<code>video</code>、<code>canvas</code> 的元素会被提升为独立图层。</p>
</li>
<li>
<p>GPU 介入，将这些图层按正确顺序组合在一起，最终显示在屏幕上。</p>
</li>
</ul>
</li>
</ol>
<h3 id="第七步断开连接四次挥手"><a href="#第七步断开连接四次挥手">第七步：断开连接（四次挥手）</a></h3>
<p>当数据传输完毕，且 HTTP 请求头中没有设置 <code>Connection: keep-alive</code>（或者达到了 Keep-Alive 的超时时间），TCP 连接会被断开。由于 TCP 是全双工的，因此需要双方分别断开自己的发送通道。</p>
<ul>
<li>
<p><strong>第一次挥手 (FIN)</strong>：主动方（通常是客户端，也可能是服务器）发送 FIN 报文，表示“我没有数据要发送了，我要断开连接”。</p>
</li>
<li>
<p><strong>第二次挥手 (ACK)</strong>：被动方收到 FIN，回复 ACK。此时主动方到被动方的方向断开，处于“半关闭”状态（被动方如果还有数据没发完，还可以继续发）。</p>
</li>
<li>
<p><strong>第三次挥手 (FIN)</strong>：被动方的数据也发送完毕了，发送 FIN 报文给主动方，表示“我也没数据要发了，准备断开”。</p>
</li>
<li>
<p><strong>第四次挥手 (ACK)</strong>：主动方收到 FIN，回复 ACK。被动方收到后立即关闭连接；主动方则进入 TIME_WAIT 状态，等待 2MSL（报文最大生存时间）后，确保对方收到了 ACK，才真正关闭连接。</p>
</li>
</ul>
