<p><strong>Q: React Fiber 架构的本质是什么？解决了什么问题？</strong></p>
<ul>
<li>
<p><strong>分析：</strong> React 16 之前的 Reconciler 是同步的、递归的。如果组件树庞大，JavaScript 线程会被长时间阻塞，导致页面掉帧、卡顿。Fiber 是一种<strong>基于时间分片的协作式多任务调度机制</strong>。它将渲染工作拆分成微小的任务单元（Fiber Node），允许 React 在执行完一个任务单元后，将控制权交还给浏览器，判断是否有更高优先级的任务（如用户点击、动画）。</p>
</li>
<li>
<p><strong>代码视角：</strong> Fiber 节点本身是一个 JavaScript 对象，构成了一个双向链表树（包含 <code>return</code>, <code>child</code>, <code>sibling</code> 指针），这使得遍历过程可以随时中断和恢复。</p>
</li>
</ul>
<p>JavaScript</p>
<pre><code class="hljs language-kotlin"><span class="hljs-comment">// 简化的 Fiber 结构</span>
<span class="hljs-keyword">const</span> fiber = {
  type: <span class="hljs-string">'div'</span>, <span class="hljs-comment">// 组件类型</span>
  props: { children: [] },
  stateNode: <span class="hljs-literal">null</span>, <span class="hljs-comment">// 对应的真实 DOM</span>
  child: <span class="hljs-literal">null</span>,     <span class="hljs-comment">// 指向第一个子节点</span>
  sibling: <span class="hljs-literal">null</span>,   <span class="hljs-comment">// 指向下一个兄弟节点</span>
  <span class="hljs-keyword">return</span>: <span class="hljs-literal">null</span>,    <span class="hljs-comment">// 指向父节点</span>
  alternate: <span class="hljs-literal">null</span>, <span class="hljs-comment">// 双缓存树（用于对比更新）</span>
};
</code></pre>
<p><strong>Q: useEffect 的闭包陷阱 (Stale Closure) 产生的原因及解法？</strong></p>
<ul>
<li>
<p><strong>分析：</strong> 这是 React 开发中最常见的问题。本质是 JavaScript 的词法作用域与 React 函数组件的执行机制碰撞的结果。每次渲染，组件内部的函数都有自己独立的闭包，捕获了当次渲染时的状态。如果 <code>useEffect</code> 的依赖数组未正确设置，其内部的定时器或异步回调引用的将永远是旧渲染帧的变量。</p>
</li>
<li>
<p><strong>代码解法：</strong> 使用依赖数组、使用函数式更新，或利用 <code>useRef</code> 保持最新值的引用。</p>
</li>
</ul>
<p>JavaScript</p>
<pre><code class="hljs language-scss">function <span class="hljs-built_in">Counter</span>() {
  const <span class="hljs-selector-attr">[count, setCount]</span> = <span class="hljs-built_in">useState</span>(<span class="hljs-number">0</span>);

  <span class="hljs-built_in">useEffect</span>(() => {
    const timer = <span class="hljs-built_in">setInterval</span>(() => {
      <span class="hljs-comment">// 错误做法：如果不加依赖，count 永远是 0</span>
      <span class="hljs-comment">// setCount(count + 1); </span>
      
      <span class="hljs-comment">// 正确做法 1：函数式更新（推荐）</span>
      <span class="hljs-built_in">setCount</span>(prev => prev + <span class="hljs-number">1</span>);
    }, <span class="hljs-number">1000</span>);
    return () => <span class="hljs-built_in">clearInterval</span>(timer);
  }, <span class="hljs-selector-attr">[]</span>); <span class="hljs-comment">// 依赖为空数组，只执行一次</span>

  return &#x3C;<span class="hljs-selector-tag">div</span>>{count}&#x3C;/<span class="hljs-selector-tag">div</span>>;
}
</code></pre>

<h4 id="vue-深度解构示例"><a href="#vue-深度解构示例">Vue 深度解构示例</a></h4>
<p><strong>Q: Vue 2 的 <code>Object.defineProperty</code> 与 Vue 3 的 <code>Proxy</code> 响应式原理深度对比？</strong></p>
<ul>
<li>
<p><strong>分析：</strong> 这是 Vue 响应式迭代的核心。Vue 2 采用 <code>defineProperty</code> 劫持对象的 getter/setter。<strong>局限性</strong>：它只能劫持已存在的属性，无法检测对象属性的添加和删除（需借用 <code>$set/$delete</code>）；对数组的索引修改和长度变化无能为力（Vue 2 内部重写了数组的 7 个变更方法来兜底）；并且需要递归遍历整个对象，对深层嵌套对象有较大的初始化性能开销。</p>
</li>
<li>
<p><strong>改进：</strong> Vue 3 采用 <code>Proxy</code>。它直接代理整个对象（而非属性），拦截对象上的多达 13 种操作（包括属性增删、原型查找等）。数组的所有操作也能天然响应。并且按需代理（只有在访问到深层属性时才包装成 Proxy），极大提升了初始化性能。</p>
</li>
</ul>

<p><strong>Q: Vue.nextTick 的底层实现逻辑？为什么优先使用微任务？</strong></p>
<ul>
<li>
<p><strong>分析：</strong> Vue 的 DOM 更新是异步的。当我们修改状态后，Vue 不会立即更新 DOM，而是将所有的状态变更推入一个队列（缓冲批处理同一事件循环中的多次修改），然后在下一个事件循环 tick 中统一执行 DOM 更新。<code>nextTick</code> 就是用于在 DOM 更新完毕后执行回调的方法。</p>
</li>
<li>
<p><strong>优先级：</strong> 浏览器事件循环中，微任务（Microtask）总是在当前宏任务（Macrotask）结束后、UI 渲染之前执行。如果用宏任务（如 <code>setTimeout</code>），会导致浏览器先渲染一次旧数据，然后再执行宏任务更新渲染新数据，造成页面闪烁。因此 Vue 优先采用 <code>Promise.then</code> 或 <code>MutationObserver</code> 等微任务实现异步队列。</p>
</li>
</ul>
