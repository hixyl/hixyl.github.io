4:T2f17,<h3 id="题目嵌套数组反转-nested-array-reversal"><a href="#题目嵌套数组反转-nested-array-reversal"><strong>题目：嵌套数组反转 (Nested Array Reversal)</strong></a></h3>
<p><strong>题目描述：</strong></p>
<p>请实现一个函数，该函数接收一个嵌套的数组作为输入。这个数组中可能包含数字以及其他数组。</p>
<p>函数的任务是，在保持原始数组的嵌套结构完全不变的情况下，将其中所有的数字进行反转。具体规则是：将所有数字按其在原数组中的“深度优先遍历”顺序提取出来，组成一个一维序列，然后将这个序列反转，最后将反转后的数字序列依次填充回原数组的结构中相应的位置。</p>
<p><strong>示例：</strong></p>
<p>给定以下输入数组：</p>
<p><strong>输入 (Input):</strong></p>
<pre><code class="hljs language-csharp">[<span class="hljs-meta">1, [2, [3, [4, 5</span>]]]]
</code></pre>
<p>该数组中的数字按出现顺序组成的序列为 [1, 2, 3, 4, 5]。</p>
<p>将这个序列反转后得到 [5, 4, 3, 2, 1]。</p>
<p>将反转后的序列填充回原有的结构中，应得到以下输出。</p>
<p><strong>期望输出 (Expected Output):</strong></p>
<pre><code class="hljs language-csharp">[<span class="hljs-meta">5, [4, [3, [2, 1</span>]]]]
</code></pre>
<h2 id="解法"><a href="#解法">解法</a></h2>
<p>这个问题非常经典，它考察的是对数组、递归以及数据结构遍历和重构的理解。从表面看是“反转”，但问题的核心并非简单地颠倒数组，而是<strong>在保持原有嵌套结构不变的前提下，将所有数值内容按顺序提取出来，反转这个数值序列，再按原结构顺序填充回去</strong>。</p>
<p>我们来一步步拆解这个问题，并给出最优的 JavaScript 实现。</p>
<h3 id="核心思路两步走的策略"><a href="#核心思路两步走的策略">核心思路：两步走的策略</a></h3>
<p>解决这个问题的最直观、最清晰的思路是“两步走”：</p>
<ol>
<li>
<p><strong>“拍平”与“提取”</strong>：首先，我们需要遍历整个嵌套数组，无论它有多深，把所有的数字（Number类型）按它们出现的顺序提取出来，放进一个一维的普通数组里。</p>
</li>
<li>
<p><strong>“反转”与“填充”</strong>：然后，我们将这个只包含数字的一维数组进行反转。最后，我们再次遍历原始的嵌套数组结构，用反转后的数字依次替换掉原来的数字。</p>
</li>
</ol>
<p>这个过程就像是把一座由不同颜色积木搭成的复杂城堡，小心地把所有积木块按顺序拆下来排成一队，然后把这一队积木的颜色顺序颠倒，最后再按原来的图纸把颜色颠倒后的积木块一个个放回原位。</p>

<h3 id="深入讲解"><a href="#深入讲解">深入讲解</a></h3>
<h4 id="1-为何选择递归"><a href="#1-为何选择递归">1. 为何选择递归？</a></h4>
<p>对于处理像树或嵌套数组这样具有不确定深度的层级结构，递归是最自然、最优雅的解决方案。</p>
<ul>
<li>
<p><strong><code>extractNumbers</code> 函数</strong>：它的任务很简单，就是“深度优先遍历”。遇到数组就往里走，遇到数字就收集起来。代码 <code>if (Array.isArray(item)) { extractNumbers(item); }</code> 正是递归的精髓，它将对当前数组的处理逻辑应用到了所有的子数组上。</p>
</li>
<li>
<p><strong><code>fillNumbers</code> 函数</strong>：它同样采用递归来重建结构。<code>map</code> 方法非常适合这个场景，因为它会遍历数组的每一项并返回一个<strong>新的</strong>数组，这保证了我们不会在修改原始数组的同时去遍历它（即“非侵入式”或“不可变性”原则）。当 <code>map</code> 遇到一个子数组时，我们不能直接复制它，而是需要递归调用 <code>fillNumbers</code> 来处理这个子数组，以确保深层的数字也被替换掉。</p>
</li>
</ul>
<h4 id="2-index-计数器的重要性"><a href="#2-index-计数器的重要性">2. <code>index</code> 计数器的重要性</a></h4>
<p>在 <code>fillNumbers</code> 函数中，我们使用了一个外部变量 <code>let index = 0;</code>。这一点至关重要。</p>
<p>我们不能在递归函数内部每次都从反转后的 <code>numbers</code> 数组中取第一个元素（例如使用 <code>numbers.shift()</code>）。为什么？</p>
<ul>
<li>
<p><strong>性能陷阱</strong>：<code>Array.prototype.shift()</code> 操作的时间复杂度是 O(N)，其中 N 是数组的剩余长度。因为它需要将删除第一个元素后的所有元素向前移动一位。如果在每次替换数字时都调用 <code>shift()</code>，整个填充过程的复杂度会飙升到 O(M2)（M为数字总数），这是非常低效的。</p>
</li>
<li>
<p><strong>逻辑正确性</strong>：使用一个共享的、在递归调用之间持续存在的 <code>index</code> 计数器，可以确保我们始终按顺序从 <code>numbers</code> 数组（<code>[5, 4, 3, 2, 1]</code>）中取值，无论递归进入了多深。它的时间复杂度仅为 O(1)，保证了整个填充过程的效率。</p>
</li>
</ul>
<h4 id="3-算法复杂度分析"><a href="#3-算法复杂度分析">3. 算法复杂度分析</a></h4>
<ul>
<li>
<p><strong>时间复杂度</strong>: O(N)。</p>
<ul>
<li>
<p>其中 N 是输入数组中所有元素（包括数字和子数组）的总数。</p>
</li>
<li>
<p><code>extractNumbers</code> 访问每个元素一次，所以是 O(N)。</p>
</li>
<li>
<p><code>numbers.reverse()</code> 的时间复杂度是 O(M)，其中 M 是数字的总数 (MleN)。</p>
</li>
<li>
<p><code>fillNumbers</code> 也访问每个元素一次来构建新数组，所以是 O(N)。</p>
</li>
<li>
<p>总时间复杂度为 O(N)+O(M)+O(N)=O(N)。算法效率是线性的，非常高效。</p>
</li>
</ul>
</li>
<li>
<p><strong>空间复杂度</strong>: O(N)。</p>
<ul>
<li>
<p><code>numbers</code> 数组需要 O(M) 的空间。</p>
</li>
<li>
<p>递归调用栈的深度取决于数组的最大嵌套层级，设为 D，需要 O(D) 的空间。</p>
</li>
<li>
<p><code>fillNumbers</code> 函数创建了一个全新的数组作为结果，这需要 O(N) 的空间。</p>
</li>
<li>
<p>综合来看，空间复杂度主要由新创建的数组决定，因此是 O(N)。</p>
</li>
</ul>
</li>
</ul>
