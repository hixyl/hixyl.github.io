4:T1b2f,<p>在前端工程化中，文件名哈希（Hash）的核心目的是<strong>解决浏览器缓存问题（Cache Busting）</strong>。通过给文件名加上基于内容计算的唯一标识，可以确保当代码更新时，文件名会随之改变，从而强迫浏览器下载最新资源，而未改变的文件则继续沿用缓存。</p>
<hr>
<h2 id="1-webpack-中的实现"><a href="#1-webpack-中的实现">1. Webpack 中的实现</a></h2>
<p>Webpack 提供了多种哈希占位符，但在生产环境中，<strong><code>[contenthash]</code></strong> 是最推荐的选择，因为它仅在文件内容发生变化时才会改变。</p>
<h3 id="配置示例"><a href="#配置示例">配置示例</a></h3>
<p>在 <code>webpack.config.js</code> 中，通过 <code>output</code> 属性进行配置：</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">const</span> path = <span class="hljs-built_in">require</span>(<span class="hljs-string">'path'</span>);
<span class="hljs-keyword">const</span> <span class="hljs-title class_">HtmlWebpackPlugin</span> = <span class="hljs-built_in">require</span>(<span class="hljs-string">'html-webpack-plugin'</span>);

<span class="hljs-variable language_">module</span>.<span class="hljs-property">exports</span> = {
  <span class="hljs-attr">entry</span>: <span class="hljs-string">'./src/index.js'</span>,
  <span class="hljs-attr">output</span>: {
    <span class="hljs-comment">// [name] 是入口名称，[contenthash] 是根据内容生成的哈希值，8位长度</span>
    <span class="hljs-attr">filename</span>: <span class="hljs-string">'[name].[contenthash:8].js'</span>, 
    <span class="hljs-attr">path</span>: path.<span class="hljs-title function_">resolve</span>(__dirname, <span class="hljs-string">'dist'</span>),
    <span class="hljs-attr">clean</span>: <span class="hljs-literal">true</span>, <span class="hljs-comment">// 每次构建前清理 dist 目录</span>
  },
  <span class="hljs-attr">plugins</span>: [
    <span class="hljs-keyword">new</span> <span class="hljs-title class_">HtmlWebpackPlugin</span>({
      <span class="hljs-attr">template</span>: <span class="hljs-string">'./src/index.html'</span>, <span class="hljs-comment">// 自动将带哈希的 js 注入到 HTML 中</span>
    }),
  ],
};
</code></pre>
<h3 id="三种哈希的区别"><a href="#三种哈希的区别">三种哈希的区别</a></h3>
<p>理解这三者的差异对于优化缓存命中率至关重要：</p>
<table>
<thead>
<tr>
<th><strong>哈希类型</strong></th>
<th><strong>作用范围</strong></th>
<th><strong>建议场景</strong></th>
</tr>
</thead>
<tbody>
<tr>
<td><strong><code>[fullhash]</code></strong></td>
<td>整个项目。只要项目中有一个文件改了，所有产物的哈希都会变。</td>
<td>开发环境（Webpack 5 之前叫 <code>hash</code>）。</td>
</tr>
<tr>
<td><strong><code>[chunkhash]</code></strong></td>
<td>基于 Webpack 的 Entry（入口）。同一个入口下的所有文件共享一个哈希。</td>
<td>简单的 JS 打包。</td>
</tr>
<tr>
<td><strong><code>[contenthash]</code></strong></td>
<td><strong>基于单个文件内容</strong>。JS 改了，CSS 没改，则 CSS 的哈希不变。</td>
<td><strong>生产环境最佳实践</strong>。</td>
</tr>
</tbody>
</table>

<h2 id="2-vite-中的实现"><a href="#2-vite-中的实现">2. Vite 中的实现</a></h2>
<p>Vite 在生产环境（<code>vite build</code>）下使用 Rollup 进行构建，<strong>默认就已经开启了文件名哈希功能</strong>。你通常不需要手动配置就能看到 <code>main.a1b2c3.js</code> 这样的文件。</p>
<p>如果你需要自定义哈希的格式，可以在 <code>vite.config.js</code> 中通过 <code>build.rollupOptions</code> 进行细粒度控制：</p>
<h3 id="配置示例-1"><a href="#配置示例-1">配置示例</a></h3>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">import</span> { defineConfig } <span class="hljs-keyword">from</span> <span class="hljs-string">'vite'</span>;

<span class="hljs-keyword">export</span> <span class="hljs-keyword">default</span> <span class="hljs-title function_">defineConfig</span>({
  <span class="hljs-attr">build</span>: {
    <span class="hljs-attr">rollupOptions</span>: {
      <span class="hljs-attr">output</span>: {
        <span class="hljs-comment">// 用于入口文件（如 index.js）</span>
        <span class="hljs-attr">entryFileNames</span>: <span class="hljs-string">'assets/[name].[hash].js'</span>,
        <span class="hljs-comment">// 用于动态导入的 chunk 文件</span>
        <span class="hljs-attr">chunkFileNames</span>: <span class="hljs-string">'assets/[name].[hash].js'</span>,
        <span class="hljs-comment">// 用于资源文件（如 css, image, font）</span>
        <span class="hljs-attr">assetFileNames</span>: <span class="hljs-string">'assets/[name].[hash].[ext]'</span>,
      },
    },
  },
});
</code></pre>
<h3 id="为什么-vite-默认就很强"><a href="#为什么-vite-默认就很强">为什么 Vite 默认就很强？</a></h3>
<p>Vite 的底层构建工具 Rollup 对内容哈希的处理非常成熟。由于 Vite 鼓励使用原生 ESM，它在构建时能更精准地分析依赖树。即便你没有配置，Vite 也会生成如下结构的产物：</p>
<ul>
<li>
<p><code>index.css</code> -> <code>index.d4e5f6.css</code></p>
</li>
<li>
<p><code>main.js</code> -> <code>main.b1c2d3.js</code></p>
</li>
</ul>
<hr>
<h2 id="3-实现逻辑的底层原理"><a href="#3-实现逻辑的底层原理">3. 实现逻辑的底层原理</a></h2>
<p>无论是 Webpack 还是 Vite，实现内容哈希的逻辑都可以解构为以下几个步骤：</p>
<ol>
<li>
<p><strong>内容读取</strong>：读取最终生成的代码字符串或二进制流。</p>
</li>
<li>
<p><strong>摘要计算</strong>：使用加密哈希算法（通常是 <strong>MD4</strong> 或 <strong>MD5</strong>，Webpack 5 默认使用更快的算法）对内容生成摘要。</p>
</li>
<li>
<p><strong>截取字符</strong>：为了文件名美观和长度限制，通常截取前 8 到 16 位。</p>
</li>
<li>
<p><strong>映射替换</strong>：将计算出的哈希值替换掉配置文件中的 <code>[hash]</code> 或 <code>[contenthash]</code> 占位符。</p>
</li>
<li>
<p><strong>HTML 注入</strong>：由于文件名变了，构建工具会自动更新 HTML 文件中的 <code>&#x3C;script src="..."></code> 和 <code>&#x3C;link href="..."></code> 标签，确保引用正确。</p>
</li>
</ol>
<blockquote>
<p><strong>注意：</strong> 在配置哈希时，一定要配合 <code>HtmlWebpackPlugin</code>（Webpack）或 Vite 默认的 HTML 处理，否则你手动写的 HTML 路径将无法找到带哈希的新文件。</p>
</blockquote>
