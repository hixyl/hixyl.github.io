<h3 id="一-网络传输阶段让资源到达得更快"><a href="#一-网络传输阶段让资源到达得更快">一、 网络传输阶段：让资源到达得更快</a></h3>
<p>这个阶段的核心思路是：<strong>请求数量越少越好，请求体积越小越好。</strong></p>
<h4 id="1-资源按需加载与代码分割-code-splitting"><a href="#1-资源按需加载与代码分割-code-splitting">1. 资源按需加载与代码分割 (Code Splitting)</a></h4>
<ul>
<li>
<p><strong>底层原理</strong>：现代前端项目通常使用 Webpack 或 Vite 等构建工具，默认会将所有 JavaScript 打包成一个巨大的文件（Bundle）。浏览器必须下载并解析完这个大文件，才能开始执行逻辑。代码分割的原理是在构建时将代码拆分成多个小块（Chunks），只有当用户访问到特定路由或触发特定交互时，才通过动态创建 <code>&#x3C;script></code> 标签去请求对应的代码块。</p>
</li>
<li>
<p><strong>典型案例</strong>：单页面应用（SPA）的首屏加载极慢，经常出现长时间的白屏。</p>
</li>
<li>

</li>

<h4 id="2-http-缓存策略"><a href="#2-http-缓存策略">2. HTTP 缓存策略</a></h4>
<ul>
<li>
<p><strong>底层原理</strong>：浏览器通过 HTTP 响应头（Headers）来决定是否缓存资源以及如何缓存。</p>
<ul>
<li>
<p><strong>强缓存</strong>（<code>Cache-Control: max-age=31536000</code>）：浏览器直接从本地读取缓存，完全不发起网络请求。适用于不常变动的静态资源（如图片、第三方库）。</p>
</li>
<li>
<p><strong>协商缓存</strong>（<code>ETag</code> / <code>Last-Modified</code>）：浏览器会向服务器发一个极小的校验请求，问“这个文件变了吗？”。如果没变，服务器返回 304 状态码，浏览器继续用本地缓存；如果变了，服务器返回 200 和新文件。</p>
</li>
</ul>
</li>
<li>
<p><strong>实现思路</strong>：前端通常配合构建工具，给生成的文件名加上内容哈希（Hash），如 <code>main.a1b2c3.js</code>。一旦代码修改，哈希改变，文件名就变了。这样我们可以放心地给所有静态资源设置超长时间的强缓存。HTML 文件不设强缓存，始终走协商缓存。</p>
</li>
</ul>
<hr>
<h3 id="二-浏览器渲染阶段减少浏览器的无用功"><a href="#二-浏览器渲染阶段减少浏览器的无用功">二、 浏览器渲染阶段：减少浏览器的无用功</a></h3>
<p>这个阶段是初级前端最容易写出性能瓶颈的地方。核心思路是：<strong>顺应浏览器的渲染机制，避免频繁打破渲染流水线。</strong></p>
<p>浏览器渲染网页的基本流程（关键渲染路径）是：</p>
<ol>
<li>
<p>解析 HTML 生成 DOM 树。</p>
</li>
<li>
<p>解析 CSS 生成 CSSOM 树。</p>
</li>
<li>
<p>将两者合并生成<strong>渲染树 (Render Tree)</strong>（剔除 <code>display: none</code> 的节点）。</p>
</li>
<li>
<p><strong>布局 (Layout / 重排)</strong>：计算每个节点在屏幕上的确切大小和位置。</p>
</li>
<li>
<p><strong>绘制 (Paint / 重绘)</strong>：将节点的颜色、阴影等视觉属性画在像素上。</p>
</li>
<li>
<p><strong>合成 (Composite)</strong>：将不同的图层合并，显示到屏幕上。</p>
</li>
</ol>
<h4 id="1-彻底理解并避免频繁的重排-reflow与重绘-repaint"><a href="#1-彻底理解并避免频繁的重排-reflow与重绘-repaint">1. 彻底理解并避免频繁的“重排 (Reflow)”与“重绘 (Repaint)”</a></h4>
<ul>
<li>
<p><strong>底层原理</strong>：</p>
<ul>
<li>
<p><strong>重绘</strong>：当你改变元素的颜色（<code>color</code>, <code>background</code>）时，浏览器只需要重新执行第 5 步（Paint）。这需要消耗性能，但尚可接受。</p>
</li>
<li>
<p><strong>重排</strong>：当你改变元素的几何属性（<code>width</code>, <code>height</code>, <code>margin</code>，或者修改 DOM 结构）时，浏览器必须回到第 4 步重新计算整个页面的布局，这往往会牵连父元素和后续兄弟元素的位置变化，随后还要重新执行绘制和合成。<strong>重排的性能开销极其昂贵。</strong></p>
</li>
</ul>
</li>
<li>
<p><strong>典型陷阱：布局抖动 (Layout Thrashing)</strong></p>
<p>浏览器很聪明，它会把多次修改样式的操作放在一个队列里，等待合适的时机批量执行。但是，如果你在用 JavaScript <strong>读取</strong>某些布局属性（如 <code>offsetTop</code>, <code>clientWidth</code>）时，浏览器为了给你返回最准确的值，会被迫立即清空队列，<strong>强制同步触发重排</strong>。</p>
</li>
<li>
<p><strong>代码案例</strong>：</p>
</li>
</ul>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-comment">// ❌ 劣质做法：在循环中交替读写布局属性，导致每次循环都触发强制重排</span>
<span class="hljs-keyword">const</span> boxes = <span class="hljs-variable language_">document</span>.<span class="hljs-title function_">querySelectorAll</span>(<span class="hljs-string">'.box'</span>);
<span class="hljs-keyword">for</span> (<span class="hljs-keyword">let</span> i = <span class="hljs-number">0</span>; i &#x3C; boxes.<span class="hljs-property">length</span>; i++) {
  <span class="hljs-comment">// 读取 offsetWidth 强制浏览器进行重排以获取最新值</span>

<h4 id="2-使用硬件加速进行动画"><a href="#2-使用硬件加速进行动画">2. 使用硬件加速进行动画</a></h4>
<ul>
<li>
<p><strong>实现思路</strong>：做动画时，尽量使用 CSS 的 <code>transform</code> 和 <code>opacity</code>，而不是修改 <code>left</code> 或 <code>top</code>。</p>
</li>
<li>
<p><strong>底层原理</strong>：修改 <code>transform</code> 和 <code>opacity</code> 不会触发重排和重绘，它们直接在第 6 步（合成 Composite）阶段由 GPU 完成处理。GPU 处理图像位移极其高效，能够保证动画达到丝滑的 60fps。</p>
</li>
</ul>
<hr>
<h3 id="三-javascript执行阶段保护主线程"><a href="#三-javascript执行阶段保护主线程">三、 JavaScript执行阶段：保护主线程</a></h3>
<ul>
<li><strong>底层原理</strong>：浏览器是多进程的，但负责执行 JavaScript 和处理页面渲染的是同一个<strong>单线程（主线程）</strong>。这意味着，如果你的 JavaScript 代码执行时间过长（比如巨大的循环计算），主线程被阻塞，浏览器就无法去响应用户的点击，也无法更新页面的绘制，页面就会表现为“卡死”。</li>
</ul>
<h4 id="1-防抖-debounce-与-节流-throttle"><a href="#1-防抖-debounce-与-节流-throttle">1. 防抖 (Debounce) 与 节流 (Throttle)</a></h4>
<p>这是前端控制高频触发事件（如 <code>scroll</code>, <code>resize</code>, <code>input</code>）的利器。</p>
<ul>
<li>
<p><strong>防抖 (Debounce) 原理</strong>：把多次高频触发合并为最后一次。就像你进电梯，只要一直有人按开门键，电梯门就不会关，直到最后一个人进来后等待几秒钟，门才关上。</p>
</li>
<li>
<p><strong>使用场景</strong>：搜索框输入联想（用户输入完毕后才发网络请求）。</p>
</li>
</ul>

<li>
<p><strong>节流 (Throttle) 原理</strong>：保证在一段时间内只执行一次。就像水龙头滴水，无论水压多大，它都只能按照固定的频率一滴一滴地落下来。</p>

</li>
</ul>

<h4 id="2-大量数据的渲染优化虚拟列表"><a href="#2-大量数据的渲染优化虚拟列表">2. 大量数据的渲染优化（虚拟列表）</a></h4>
<ul>
<li>
<p><strong>底层原理</strong>：如果后端一次性返回一万条数据，前端直接生成一万个 DOM 节点插入页面，浏览器瞬间就会崩溃。因为 DOM 节点是非常庞大的对象，创建和渲染成千上万个节点极其消耗内存和主线程算力。</p>
</li>
<li>
<p><strong>实现思路</strong>：虚拟列表（Virtual List）。无论数据有多少条，只渲染屏幕可视区域内的几十个 DOM 节点。当用户滚动时，通过计算滚动距离，动态替换这几十个 DOM 节点内的数据内容，并利用 <code>transform: translateY</code> 调整它们的位置。</p>
</li>
</ul>
