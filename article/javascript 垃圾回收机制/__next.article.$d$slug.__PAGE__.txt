4:T5c39,<p>我们来系统且深入地探讨一下 JavaScript 的垃圾回收（Garbage Collection, GC）机制，并结合实际案例，说明如何利用这些知识来优化代码性能。</p>
<h3 id="核心思想为何需要垃圾回收"><a href="#核心思想为何需要垃圾回收">核心思想：为何需要垃圾回收？</a></h3>
<p>从第一性原理出发，程序运行的本质是处理数据，而数据需要存储在内存中。内存是有限的资源。如果程序只申请内存而不释放，内存最终会被耗尽，导致程序崩溃。</p>
<p>在像 C/C++ 这样的语言中，开发者需要手动管理内存，通过 <code>malloc()</code> 申请，通过 <code>free()</code> 释放。这种方式给予了开发者极大的控制权，但同时也带来了巨大的心智负担和风险——忘记释放会导致<strong>内存泄漏（Memory Leak）</strong>，而释放了仍在使用的内存则会导致**悬挂指针（Dangling Pointer）**和程序崩溃。</p>
<p>JavaScript 的设计哲学之一是让开发者更专注于业务逻辑，而非底层细节。因此，它采用了<strong>自动垃圾回收机制</strong>。这意味着，开发者只管申请和使用内存（例如，创建一个对象 <code>let a = {}</code>），而“哪些内存不再需要”以及“如何释放它们”则由 JavaScript 引擎自动完成。</p>
<p><strong>垃圾回收的核心任务</strong>：识别并回收不再被“需要”的内存。</p>
<h3 id="js-垃圾回收的基石可达性reachability"><a href="#js-垃圾回收的基石可达性reachability">JS 垃圾回收的基石：可达性（Reachability）</a></h3>
<p>现代垃圾回收算法的根基是<strong>可达性</strong>这个概念。简单来说，一个对象是否“存活”，取决于它是否可以从一个“根（Root）”对象沿着引用链被访问到。</p>
<ul>
<li>
<p><strong>根（Roots）</strong>：这是一组固定的、总是可访问的引用，它们是可达性分析的起点。主要包括：</p>
<ul>
<li>
<p><strong>全局对象</strong>：比如浏览器中的 <code>window</code> 对象，Node.js 中的 <code>global</code> 对象。</p>
</li>
<li>
<p><strong>当前调用栈</strong>：函数调用时，其中的局部变量、参数等。</p>
</li>
<li>
<p><strong>CPU 寄存器</strong>中的值。</p>
</li>
</ul>
</li>
</ul>
<p><strong>回收过程的本质</strong>：从所有的“根”出发，遍历所有可访问的对象。任何遍历结束后未能访问到的对象，就被认为是“垃圾”，可以被回收。</p>
<p><em>在上图中，从 Root 出发，对象 C 和 E 是不可达的，因此它们是垃圾，将被回收。</em></p>
<hr>
<h3 id="主要的垃圾回收算法"><a href="#主要的垃圾回收算法">主要的垃圾回收算法</a></h3>
<h4 id="1-引用计数reference-counting--已被淘汰"><a href="#1-引用计数reference-counting--已被淘汰">1. 引用计数（Reference Counting）- 已被淘汰</a></h4>
<p>这是早期的一种简单算法，但如今主流的 JS 引擎已不再使用它作为核心算法。</p>
<ul>
<li>
<p><strong>原理</strong>：为每个对象维护一个“引用计数器”。当有一个引用指向该对象时，计数器加 1；当引用被移除时，计数器减 1。当计数器变为 0 时，说明没有任何地方引用该对象，可以立即回收。</p>
</li>
<li>
<p><strong>致命缺陷</strong>：<strong>循环引用（Circular References）</strong>。</p>
</li>
</ul>
<p>看一个例子：</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">function</span> <span class="hljs-title function_">createCircularReference</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">let</span> objectA = {};
    <span class="hljs-keyword">let</span> objectB = {};

    objectA.<span class="hljs-property">b</span> = objectB; <span class="hljs-comment">// objectB 的引用计数为 1</span>
    objectB.<span class="hljs-property">a</span> = objectA; <span class="hljs-comment">// objectA 的引用计数为 1</span>

    <span class="hljs-comment">// 函数执行完毕后，objectA 和 objectB 的引用都消失了</span>
    <span class="hljs-comment">// 但是，它们内部互相引用，导致各自的引用计数永远不会是 0</span>
}

<span class="hljs-title function_">createCircularReference</span>();
<span class="hljs-comment">// objectA 和 objectB 无法被回收，造成内存泄漏。</span>
</code></pre>
<p>由于这个致命缺陷，现代 JS 引擎转向了下面更复杂的算法。</p>
<h4 id="2-标记-清除mark-and-sweep---现代gc的基础"><a href="#2-标记-清除mark-and-sweep---现代gc的基础">2. 标记-清除（Mark-and-Sweep） - 现代GC的基础</a></h4>
<p>这是现代垃圾回收算法的核心思想，它完美地解决了循环引用的问题。</p>
<ul>
<li>
<p><strong>原理</strong>：分为两个阶段。</p>
<ol>
<li>
<p><strong>标记阶段（Mark）</strong>：GC 从“根”开始，递归地遍历所有可达的对象，并给它们打上“存活”的标记。</p>
</li>
<li>
<p><strong>清除阶段（Sweep）</strong>：GC 遍历整个内存堆（Heap），回收所有没有被标记的对象。</p>
</li>
</ol>
</li>
</ul>
<p>这个算法可以正确处理循环引用，因为即使 <code>objectA</code> 和 <code>objectB</code> 互相引用，但如果它们从“根”出发是不可达的，它们就不会在标记阶段被标记，最终会在清除阶段被回收。</p>
<ul>
<li><strong>缺点</strong>：<strong>内存碎片化（Memory Fragmentation）</strong>。清除后，内存中会留下许多不连续的空闲空间。当需要分配一个较大的对象时，可能因为找不到足够大的连续空间而失败，尽管总的空闲空间是足够的。</li>
</ul>
<h4 id="3-标记-整理mark-and-compact"><a href="#3-标记-整理mark-and-compact">3. 标记-整理（Mark-and-Compact）</a></h4>
<p>为了解决内存碎片化问题，标记-整理算法在标记-清除的基础上增加了一个步骤。</p>
<ul>
<li>
<p><strong>原理</strong>：</p>
<ol>
<li>
<p><strong>标记阶段（Mark）</strong>：同上，标记所有存活对象。</p>
</li>
<li>
<p><strong>整理阶段（Compact）</strong>：将所有存活的对象向内存的一端移动，使它们紧凑地排列在一起。</p>
</li>
<li>
<p><strong>清除阶段（Sweep）</strong>：直接清理掉整理边界之后的所有内存。</p>
</li>
</ol>
</li>
</ul>
<p>这样，不仅回收了垃圾，还消除了碎片，让后续的内存分配变得简单高效。</p>
<h3 id="v8-引擎的优化分代回收generational-collection"><a href="#v8-引擎的优化分代回收generational-collection">V8 引擎的优化：分代回收（Generational Collection）</a></h3>
<p>现代 JS 引擎（如 Chrome 的 V8）为了极致的性能，并不会简单地对所有对象都使用同一种算法。它们基于一个重要的观察——<strong>“分代假说”（The Generational Hypothesis）</strong>：</p>
<blockquote>
<p>大多数对象在内存中存在的时间很短，很快就会变成垃圾。只有少数对象会存活很长时间。</p>
</blockquote>
<p>基于此，V8 将内存堆分为了两个主要区域：</p>
<ol>
<li>
<p><strong>新生代（New Space / Young Generation）</strong>：</p>
<ul>
<li>
<p><strong>特点</strong>：容量小（通常几MB），存放新创建的、生命周期短的对象。</p>
</li>
<li>
<p><strong>垃圾回收方式（Minor GC / Scavenge）</strong>：非常频繁且快速。新生代内部又分为两个等大的空间：From-Space 和 To-Space。</p>
<ul>
<li>
<p>新对象被分配在 From-Space。</p>
</li>
<li>
<p>当 From-Space 满了，触发 Minor GC。</p>
</li>
<li>
<p>GC 检查 From-Space 中的存活对象，并将它们<strong>复制</strong>到 To-Space。</p>
</li>
<li>
<p>复制完成后，From-Space 和 To-Space 的角色互换。</p>
</li>
<li>
<p>这个过程天然地完成了内存整理，不会产生碎片。</p>
</li>
</ul>
</li>
<li>
<p><strong>晋升（Promotion）</strong>：如果一个对象在多次 Minor GC 后依然存活，它就会被“晋升”到老生代。</p>
</li>
</ul>
</li>
<li>
<p><strong>老生代（Old Space / Old Generation）</strong>：</p>
<ul>
<li>
<p><strong>特点</strong>：容量大，存放生命周期长的对象（例如，全局变量、存活很久的闭包）以及从新生代晋升上来的对象。</p>
</li>
<li>
<p><strong>垃圾回收方式（Major GC）</strong>：不那么频繁，但更耗时。主要采用**标记-清除（Mark-Sweep）<strong>和</strong>标记-整理（Mark-Compact）**相结合的方式。</p>
</li>
<li>
<p><strong>优化</strong>：为了避免长时间的 Major GC 导致页面卡顿（“Jank”），V8 引入了**增量标记（Incremental Marking）<strong>和</strong>并发标记（Concurrent Marking）**等技术，将标记工作拆分成许多小任务，穿插在 JavaScript 执行的间隙中完成，从而减少主线程的阻塞时间。</p>
</li>
</ul>
</li>
</ol>

<h3 id="利用垃圾回收机制优化性能的常见案例"><a href="#利用垃圾回收机制优化性能的常见案例">利用垃圾回收机制优化性能的常见案例</a></h3>
<p>理解了 GC 原理，我们就能写出对 GC 更友好的代码，从而提升应用性能。核心思路是：<strong>减少 GC 的触发频率和单次 GC 的开销</strong>。</p>
<h4 id="案例一警惕全局变量使用作用域管理内存"><a href="#案例一警惕全局变量使用作用域管理内存">案例一：警惕全局变量，使用作用域管理内存</a></h4>
<ul>
<li>
<p><strong>原理</strong>：全局变量是“根”的一部分，除非手动设置为 <code>null</code>，否则它们引用的对象永远不会被回收，会一直存在于<strong>老生代</strong>中。</p>
</li>
<li>
<p><strong>不良实践</strong>：</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-comment">// 假设这是某个数据获取模块</span>
<span class="hljs-keyword">let</span> largeDataCache = []; <span class="hljs-comment">// 全局缓存</span>

<span class="hljs-keyword">function</span> <span class="hljs-title function_">fetchData</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">const</span> data = <span class="hljs-title function_">getSomeVeryLargeData</span>(); <span class="hljs-comment">// 获取了 10MB 的数据</span>
    largeDataCache.<span class="hljs-title function_">push</span>(...data);
    <span class="hljs-comment">// largeDataCache 会持续增长，永远不会被回收</span>
}
</code></pre>
</li>
<li>
<p><strong>优化方案</strong>：<strong>使用局部作用域</strong>。让变量在不需要时能够自然地脱离作用域，从而被 GC 回收。</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">function</span> <span class="hljs-title function_">processData</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">const</span> data = <span class="hljs-title function_">getSomeVeryLargeData</span>(); <span class="hljs-comment">// data 是局部变量</span>
    <span class="hljs-comment">// ...对 data 进行处理...</span>
} <span class="hljs-comment">// 函数执行完毕后，data 引用的对象就变得不可达，可以被回收了</span>

<span class="hljs-comment">// 如果确实需要缓存，应提供明确的清理机制</span>
<span class="hljs-keyword">class</span> <span class="hljs-title class_">DataCache</span> {
    <span class="hljs-title function_">constructor</span>(<span class="hljs-params"></span>) { <span class="hljs-variable language_">this</span>.<span class="hljs-property">cache</span> = []; }
    <span class="hljs-title function_">fetch</span>(<span class="hljs-params"></span>) { <span class="hljs-comment">/* ... */</span> }
    <span class="hljs-title function_">clear</span>(<span class="hljs-params"></span>) { <span class="hljs-variable language_">this</span>.<span class="hljs-property">cache</span> = []; } <span class="hljs-comment">// 手动释放</span>
}
</code></pre>
<p><strong>最佳选择</strong>：优先使用函数/块级作用域内的 <code>const</code> 和 <code>let</code>，避免使用 <code>var</code> 和不必要的全局变量。这能确保变量的生命周期最短，使其更有可能在<strong>新生代</strong>就被快速回收。</p>
</li>
</ul>
<h4 id="案例二小心闭包closure造成的意外内存泄漏"><a href="#案例二小心闭包closure造成的意外内存泄漏">案例二：小心闭包（Closure）造成的意外内存泄漏</a></h4>
<ul>
<li>
<p><strong>原理</strong>：闭包会使其内部函数持有对外部作用域变量的引用。如果这个内部函数变成了长生命周期的对象（如事件监听器、定时器），那么它引用的外部变量也无法被回收。</p>
</li>
<li>
<p><strong>不良实践</strong>：</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">function</span> <span class="hljs-title function_">attachListener</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">let</span> largeObject = <span class="hljs-keyword">new</span> <span class="hljs-title class_">Array</span>(<span class="hljs-number">1e6</span>).<span class="hljs-title function_">fill</span>(<span class="hljs-string">'*'</span>); <span class="hljs-comment">// 一个很大的对象</span>
    <span class="hljs-keyword">let</span> element = <span class="hljs-variable language_">document</span>.<span class="hljs-title function_">getElementById</span>(<span class="hljs-string">'myButton'</span>);

    <span class="hljs-comment">// click 回调函数是一个闭包，它引用了 largeObject</span>
    element.<span class="hljs-title function_">addEventListener</span>(<span class="hljs-string">'click'</span>, <span class="hljs-keyword">function</span> <span class="hljs-title function_">onClick</span>(<span class="hljs-params"></span>) {
        <span class="hljs-comment">// ... 也许这里并没有用到 largeObject</span>
        <span class="hljs-variable language_">console</span>.<span class="hljs-title function_">log</span>(<span class="hljs-string">'Button clicked!'</span>);
    });

    <span class="hljs-comment">// largeObject = null; // 如果不手动解除引用...</span>
}
<span class="hljs-title function_">attachListener</span>();
<span class="hljs-comment">// 只要 #myButton 元素存在，它的 click 监听器就存在</span>
<span class="hljs-comment">// 监听器存在，就一直引用着 largeObject</span>
<span class="hljs-comment">// 导致 largeObject 这个大对象被意外地留在了老生代中，无法回收</span>
</code></pre>
</li>
<li>
<p><strong>优化方案</strong>：在不再需要时，主动解除引用或移除事件监听器。</p>
<ol>
<li>
<p><strong>主动解除引用</strong>：如果闭包内部不再需要某个外部变量，在合适的时机（例如，组件卸载时）将其设置为 <code>null</code>。</p>
</li>
<li>
<p><strong>移除监听器</strong>：在 DOM 元素被销毁或不再需要监听时，使用 <code>removeEventListener</code> 移除监听器。这会自动断开闭包的引用链。</p>
</li>
</ol>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-keyword">function</span> <span class="hljs-title function_">setupComponent</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">let</span> largeObject = <span class="hljs-keyword">new</span> <span class="hljs-title class_">Array</span>(<span class="hljs-number">1e6</span>).<span class="hljs-title function_">fill</span>(<span class="hljs-string">'*'</span>);
    <span class="hljs-keyword">let</span> element = <span class="hljs-variable language_">document</span>.<span class="hljs-title function_">getElementById</span>(<span class="hljs-string">'myButton'</span>);

    <span class="hljs-keyword">function</span> <span class="hljs-title function_">onClick</span>(<span class="hljs-params"></span>) {
        <span class="hljs-variable language_">console</span>.<span class="hljs-title function_">log</span>(<span class="hljs-string">'Button clicked!'</span>);
    }
    element.<span class="hljs-title function_">addEventListener</span>(<span class="hljs-string">'click'</span>, onClick);

    <span class="hljs-comment">// 返回一个清理函数，这是 React/Vue 等框架的常见模式</span>
    <span class="hljs-keyword">return</span> <span class="hljs-keyword">function</span> <span class="hljs-title function_">cleanup</span>(<span class="hljs-params"></span>) {
        element.<span class="hljs-title function_">removeEventListener</span>(<span class="hljs-string">'click'</span>, onClick);
        <span class="hljs-comment">// 监听器被移除，闭包被销毁，largeObject 变得不可达</span>
        largeObject = <span class="hljs-literal">null</span>; <span class="hljs-comment">// 显式设置更好</span>
    };
}

<span class="hljs-keyword">const</span> cleanup = <span class="hljs-title function_">setupComponent</span>();
<span class="hljs-comment">// ...在某个时刻，比如页面切换时</span>
<span class="hljs-title function_">cleanup</span>();
</code></pre>
</li>
</ul>
<h4 id="案例三对象数组池化objectarray-pooling"><a href="#案例三对象数组池化objectarray-pooling">案例三：对象/数组池化（Object/Array Pooling）</a></h4>
<ul>
<li>
<p><strong>原理</strong>：对于需要频繁创建和销毁的同类对象（例如，游戏中的子弹、粒子效果，或数据可视化中的几何体对象），反复创建会导致新生代频繁的 Minor GC，增加 CPU 开销。</p>
</li>
<li>
<p><strong>不良实践</strong>：</p>
<p>JavaScript</p>
<pre><code class="hljs language-js"><span class="hljs-comment">// 动画循环中</span>
<span class="hljs-keyword">function</span> <span class="hljs-title function_">gameLoop</span>(<span class="hljs-params"></span>) {
    <span class="hljs-keyword">for</span> (<span class="hljs-keyword">let</span> i = <span class="hljs-number">0</span>; i &#x3C; <span class="hljs-number">100</span>; i++) {
        <span class="hljs-keyword">let</span> particle = { <span class="hljs-attr">x</span>: <span class="hljs-number">0</span>, <span class="hljs-attr">y</span>: <span class="hljs-number">0</span>, <span class="hljs-attr">life</span>: <span class="hljs-number">10</span> }; <span class="hljs-comment">// 每帧创建 100 个新对象</span>
        <span class="hljs-comment">// ...更新粒子状态...</span>
    }
    <span class="hljs-comment">// 每一帧都会产生大量立即被丢弃的对象，加重 GC 负担</span>

</li>
<li>

</li>
</ul>
<h3 id="总结"><a href="#总结">总结</a></h3>
<p>JavaScript 的垃圾回收机制是其易用性的重要保障，但它并非“免费的午餐”。其运行本身会消耗计算资源，并在某些情况下阻塞主线程。</p>
<p>作为开发者，虽然我们不能直接控制 GC，但通过深入理解其工作原理——尤其是<strong>可达性</strong>和<strong>分代回收</strong>——我们可以编写出更加高效、对 GC 更友好的代码：</p>
<ol>
<li>
<p><strong>最小化内存占用和生命周期</strong>：优先使用局部变量，让不再需要的对象尽快变得不可达。</p>
</li>
<li>
<p><strong>警惕长生命周期的引用</strong>：注意全局变量、闭包和事件监听器等可能意外延长对象生命周期的模式。</p>
</li>
<li>
<p><strong>复用代替创建</strong>：在性能敏感的循环中，考虑使用对象池等技术来减少垃圾的产生。</p>
</li>
</ol>
